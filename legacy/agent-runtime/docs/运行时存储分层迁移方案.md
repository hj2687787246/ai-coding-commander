# 运行时存储分层迁移方案

更新时间：2026-04-12

## 1. 目标

把当前仓库里的 `.runtime/` 从“混合主存储 + 调试目录”收成明确分层：

1. Redis：短生命周期运行态
2. SQL：长期可查询元数据
3. 文件 / 对象存储：大块原始产物
4. `.runtime/`：本地缓存、调试残留、开发期临时产物

一句话目标：

**让 `.runtime/` 从主账本降级为本地运行时目录，而不是继续承担多用户、多会话场景下的长期主存储职责。**

## 3. 分层原则

### 3.1 Redis 层

只放短 TTL、适合按 key 读写、适合并发协调的运行态：

1. request execution summary
2. session recent request execution index
3. token 吊销
4. shortcut cache
5. commander 当前 handoff / 活跃任务瞬时状态

### 3.2 SQL 层

只放长期要查询、要筛选、要统计的结构化元数据：

1. session 基本信息
2. message 索引、摘要、统计字段
3. evaluation job 元数据
4. commander task / checkpoint / report 元数据
5. improvement candidate / skill candidate 元数据

### 3.3 文件 / 对象存储层

保留大块正文和原始产物，不强行全塞数据库：

1. chat 原始消息正文
2. meta / state 快照
3. evaluation 报告 JSON / Markdown
4. retrieval trace 原文
5. commander packet / worker_brief / worker_report / report 快照
6. skill candidate 草稿正文

### 3.4 `.runtime/` 层

只保留开发机临时目录、缓存和调试残留：

1. pid
2. stdout / stderr log
3. manual smoke 产物
4. pytest / 临时调试目录
5. 允许被安全清理的本地残留

## 4. 迁移边界

### 4.1 本轮要做

1. 明确每个 `.runtime` 子目录的目标归属
2. 先补索引和元数据，不先推翻现有恢复链
3. 让新写入优先走分层目标，旧数据保留兼容期
4. 最终把 `.runtime` 降级成缓存 / 调试层

### 4.2 本轮不做

1. 不一次性把所有 JSON 改成数据库表
2. 不把 trace / 报告 / candidate 正文全部塞进 SQL
3. 不先做对象存储平台化
4. 不顺手重写 chat 恢复逻辑和 commander harness 逻辑

### Phase C 第一刀：指挥官 harness 分层收口

这一步先不把 `commander/tasks/*` 迁成数据库主账本，而是先把“哪些文件负责真相源、哪些字段负责查询、哪些内容继续留快照”写死，避免下一刀再回头争论。

#### 1. `commander/tasks/*` 里的文件各自承担什么角色

当前 harness 已经把 task 目录拆成了比较清楚的几类文件：

1. `packet.json`
   - 任务派发契约
   - 记录 `task_id / title / goal / must_read / bounds / validation / forbidden_paths / worker_profile / tool_profile / allowed_tools / reuse_allowed / report_contract / status`
2. `worker_brief.md`
   - 给执行窗口看的可读版任务简报
   - 重点是把 packet 的约束翻译成易执行的说明，不承担查询账本职责
3. `worker_report.json`
   - 执行窗口回填的真实结果
   - 承载 `status / summary / changed_files / verification / commit / risks / recommended_next_step / needs_commander_decision` 等内容
4. `report.json`
   - 指挥官 ingest 之后的正式结果快照
   - 作为 `resume / status` 的稳定读源之一
5. `reports/*.json`
   - 同一份 report 的归档快照
   - 保留历史痕迹，不参与在线更新
6. `checkpoint.json`
   - 恢复锚点
   - 记录 `current_phase / recommended_action / next_minimal_action / controller_handoff / active_subagents / blockers / pending_decisions / key_paths / updated_at`
7. `status.json`
   - 任务当前状态摘要
   - 由 `refresh_status()` 计算，供 `commander_status.py` 直接读取
8. `events.jsonl`
   - 追加式事件日志
   - 记录派发、 ingest、状态刷新等时间线事实，不做覆盖式写入
9. `.runtime/commander/workers/slots/*.json` 与 `registry.json`
   - worker pool 的运行态，不是 task 历史本体
   - 只负责 worker 复用和占用状态

`commander_harness.py` 里的现状也很明确：`refresh_status()` 会同时读 `packet.json`、`report.json`、`events.jsonl`，再把结果回写到 `status.json` 和 `checkpoint.json`；`commander_resume.py` 只负责读 checkpoint；`commander_status.py` 只负责读当前 snapshot；`commander_ingest.py` 负责把报告落成正式 `report.json` 并追加事件。也就是说，当前真实主路径仍然是“文件快照 + 事件日志”，不是数据库。

#### 2. 哪些字段应该进入可查询元数据层的最小字段集

Phase C 第一刀只收最小查询索引，不把 report 正文和 checkpoint 细节全拆进列。建议先收这些字段：

1. `task_id`
2. `title`
3. `current_phase`
4. `status`
5. `worker_profile`
6. `preferred_worker_profile`
7. `tool_profile`
8. `allowed_tools`
9. `controller_handoff`
10. `commander_recommendation`
11. `recommended_action`
12. `next_minimal_action`
13. `worker_status`
14. `needs_commander_decision`
15. `needs_user_decision`
16. `ready_for_user_delivery`
17. `has_packet`
18. `has_report`
19. `event_count`
20. `last_event_type`
21. `last_event_at`
22. `updated_at`

如果后面要做 task 列表、筛选、恢复入口，这组字段已经够用了：

1. 能按 task 识别和排序
2. 能按 worker / tool profile 过滤
3. 能按控制面转向判断任务是否该继续、等待外部结果，还是收口
4. 能直接判断是否已经有 report

#### 3. 哪些内容继续留在文件快照 / 事件日志里

第一刀不要急着把这些内容数据库化：

1. `packet.json` 里的完整派发契约正文
2. `worker_brief.md` 的人类可读说明
3. `worker_report.json` 的完整执行结果、`changed_files`、`verification`、`commit`、`risks`、`recommended_next_step`
4. `checkpoint.json` 的恢复细节，尤其是 `active_subagents`、`blockers`、`pending_decisions`、`key_paths`
5. `events.jsonl` 的逐条时间线
6. `reports/*.json` 的历史归档快照
7. worker pool 的 `slots/*.json` 和 `registry.json`

原因是这些内容要么是大块正文，要么是恢复细节，要么是 append-only 历史。第一刀如果先把它们展开成列，成本高、收益低，而且会把恢复语义拆散。

#### 4. 第一承接层应该放哪一层

结合当前仓库现状，Phase C 第一刀更适合先放在 `别的层`，也就是继续以 `.runtime/commander/tasks/*` 的文件快照和 `events.jsonl` 作为第一承接层，而不是先切到现有 SQLite。

判断依据很直接：

1. 现在 `commander_harness.py / commander_dispatch.py / commander_ingest.py / commander_status.py / commander_resume.py` 全部都是文件驱动，没有 commander 专用的 SQLite 读写链路
2. 仓库里的主数据库目前也没有 commander task 的实体模型或迁移入口可直接承接
3. `evaluation_jobs.db` 是评估链路遗留的本地 SQLite，不是 commander harness 的自然归属，硬接会把两条不同职责的运行态混在一起
4. 如果现在再引入一层新的 commander SQLite 过渡层，只会多一套同步和回填路径，反而拖慢收口

所以这一刀的顺序应该是：

1. 先把 task 目录里的文件角色和最小索引字段定死
2. 再把可查询元数据层接到主数据库，而不是先造本地 SQLite 过渡账本
3. 维持文件快照和事件日志继续做真相源，等索引层稳定后再考虑是否拆更细的表

#### 5. 第二实现切片的主数据库承接面

这一刀已经把第一版可查询索引表落到了主数据库：`commander_task_catalog` 只承接 catalog 的最小字段面，不接 `report.json` 正文和 `events.jsonl` 全量明细。

同步原则保持不变：

1. 文件快照和事件日志仍是真相源
2. `commander_task_catalog` 只负责查询摘要
3. 后续如果要再拆更细的结构化索引，也是在这张表的字段面上渐进演进

## 9. Phase B 第一刀：会话恢复分层收口

这一步不再讨论“要不要上数据库”，而是基于当前实现把会话恢复的真相源边界写死。

### 9.1 `chat_history` 进入 SQL 的最小 session 元数据字段集

第一刀只把“侧边栏列表、会话恢复、归属校验”真正依赖的字段收进主数据库索引层，不额外扩张 schema：

1. `session_id`
2. `owner_user_id`
3. `owner_username`
4. `owner_display_name`
5. `session_title`
6. `preview`
7. `pinned`
8. `message_count`
9. `created_at`
10. `updated_at`

这里的判断标准很直接：

1. `api/routes/session.py` 里已经用这些字段做了列表展示、排序、归属校验和详情恢复
2. `utils/session_index.py` 里 `resolve_session_preview / resolve_session_message_count / touch_session_metadata_index` 也只围绕这组字段工作
3. `chat_messages` 负责正文，`chat_sessions` 负责可查询摘要，不需要再为第一刀额外拆一张“过渡索引表”

### 9.2 已经由主数据库 `ChatSession / ChatSessionState` 承接的字段

主数据库其实已经把第一刀最需要的东西接住了，现状不是“没模型”，而是“边界还没写进文档和后续任务”。

1. `ChatSession`
   - `session_id`
   - `owner_user_id`
   - `owner_username`
   - `owner_display_name`
   - `session_title`
   - `preview`
   - `pinned`
   - `message_count`
   - `created_at`
   - `updated_at`
2. `ChatSessionState`
   - `metadata_json`
   - `state_json`
   - `created_at`
   - `updated_at`

对应到当前代码：

1. `rag/history_store.py` 里的 `set_session_metadata()` 会把会话摘要同步进 `ChatSession`
2. `rag/history_store.py` 里的 `set_session_state()` / `update_session_state()` 会把跨轮状态放进 `ChatSessionState.state_json`
3. `rag/history_store.py` 里的 `get_session_metadata()` 仍然负责读 `ChatSessionState.metadata_json`，给兼容期保留额外字段

### 9.3 暂时仍留在文件正文或 `metadata_json` 的字段

第一刀不把所有会话附属信息硬塞进 `ChatSession` 列；下面这些先留在 `metadata_json`，旧数据迁移时则仍可从 `messages.json / meta.json / state.json` 读出来：

1. `last_user_query`
2. `reference_markdowns`
3. `last_reference_markdown`
4. `thought_summaries`
5. `status_histories`
6. 其它按 assistant 轮次增长、偏展示用途的变长数组

原因很简单：

1. 这些字段是“按轮次累积”的展示附属信息，不是列表页必须的索引字段
2. `api/routes/session.py` 目前就是靠 `metadata_json` 把引用区、思考区、状态区按轮次插回消息流
3. 这些内容更像恢复增强数据，不适合在第一刀就展开成更多列，避免把一个小收口做成大表重构

对文件正文的处理口径也要说清楚：

1. 新写入主路径已经在 `chat_messages`，不再把正文当作 `.runtime/chat_history` 的主账本
2. `scripts/migrate_storage_to_primary_db.py` 仍保留对旧 `messages.json / meta.json / state.json` 的导入兼容
3. 只有在迁移和回填场景里，文件正文才继续作为过渡来源

### 9.4 第一承接层为什么优先放主数据库，而不是本地 SQLite 过渡层

优先主数据库，不优先本地 SQLite 过渡层，原因不是“偏好”，而是当前实现已经把路走死在主库上了：

1. `rag/history_store.py` 的 `get_history / set_session_metadata / set_session_state` 已经直接通过 `api/db.py` 的 `session_scope()` 进主库事务
2. `api/routes/session.py` 的列表、更新、读取、删除都依赖 `ChatSession / ChatSessionState`，没有再单独维护本地 SQLite 的读写分支
3. `scripts/migrate_storage_to_primary_db.py` 也已经把旧文件会话和旧 SQLite 评估数据往主库收，方向是“收口到主库”，不是再生一层本地过渡库
4. 如果再加一层本地 SQLite，就会把当前已经存在的“文件 + 运行时元数据合并”再升级成“三套来源”，双写和回填成本只会更高
5. 主数据库已经承担了归属、列表、排序、消息计数和状态恢复的最小闭环，本地 SQLite 在这一刀里不提供新的能力，只会延长迁移路径

### 9.5 Phase B 第一刀的最小实施顺序

只针对 Phase B 第一刀，建议按下面顺序推进，不要一上来拆大工程：

1. 先确认会话列表和恢复需要的最小字段全部由 `ChatSession / ChatSessionState` 提供，不再依赖目录扫描做主路径
2. 再把 `message_count / updated_at / session_title / preview / pinned` 这组摘要字段的写入和更新收敛到主库同步路径
3. 保留 `ChatSessionState.metadata_json` 承载 `last_user_query / reference_markdowns / thought_summaries / status_histories`，不急着拆列
4. 旧 `.runtime/chat_history` 只保留迁移导入兼容，不作为新写入主路径
5. 等第一刀收口完成，再决定是否需要第二刀把 `metadata_json` 继续细分成更稳定的结构化列
## 10. Phase D 第一刀：评估产物分层收口

这一刀的目标不是再新建一套评估账本，而是把 `.runtime/rag_evaluation/` 和 `.runtime/rag_answer_evaluation/` 收成“文件原型 + SQL 元数据层”的组合：先让历史报告能按 job / case 查询，再保留正文和 trace 作为真相源。

### 10.1 `rag_evaluation` / `rag_answer_evaluation` 现在各自承担什么

1. `.runtime/rag_evaluation/*`
   - retrieval 评估的报告与 trace 真相源
   - 真实查询入口是 `evaluate_rag.py`，报告正文和 case 级 trace 都是文件产物
2. `.runtime/rag_answer_evaluation/*`
   - answer-level 评估的报告真相源
   - 真实查询入口是 `evaluate_rag_answers.py`，Markdown 负责可读正文，JSON 负责结构化结果
3. `EvaluationJob`
   - 现在已经是仓库里现成的 job 级元数据模型
   - 字段天然适合承接“这次评估跑了什么、跑到哪、结果文件在哪”

### 10.2 最小可查询元数据字段集

Phase D 第一刀先把“能筛选、能排序、能回到文件”的字段放进元数据层，不把报告正文拆成列：

1. `job_id`
2. `evaluation_type`
3. `evaluation_label`
4. `status`
5. `stage`
6. `stage_label`
7. `dataset_path`
8. `created_at`
9. `started_at`
10. `updated_at`
11. `finished_at`
12. `current_case_index`
13. `current_case_id`
14. `current_case_query`
15. `completed_cases`
16. `total_cases`
17. `max_concurrency`
18. `report_json_path`
19. `report_markdown_path`
20. `trace_paths_json`
21. `source_job_id`
22. `request_payload_json`
23. `instance_id`
24. `coordination_token`
25. `heartbeat_at`
26. `lease_expires_at`

这组字段已经足够做三件事：

1. 按时间、类型、状态、阶段筛评估任务
2. 直接跳回对应的 report JSON / Markdown / trace 文件
3. 在列表页或检索页上看到 job 级进度，而不用展开全量正文

### 10.3 哪些内容继续留在文件里

第一刀不要把这些内容列化：

1. retrieval 报告里的 `final_results / reranked_results / dense_results / sparse_results`
2. retrieval 报告里的 `processed_query / search_plan / timings / reference_decision / metrics` 细节
3. answer-level 报告里的完整 Markdown 正文
4. answer-level 报告里的逐 case 结果数组、judge 解释、上下文列表和原始引用片段
5. trace 文件本身，以及任何 prompt / model / chain 逐步执行细节

原因很直接：

1. 这些内容更像“可回放证据”，不是 job 列表页必须字段
2. 全量展开会把一份报告拆成很多不稳定列，后续 schema 维护成本高
3. 真正需要搜索的，是“哪次评估、什么类型、什么状态、报告在哪”，不是把每个 chunk 都变成表列

### 10.4 第一承接层放哪里

结论：第一承接层继续放主数据库里现成的 `evaluation_jobs`，不要先再造一层新的本地 SQLite 过渡表。

判断依据是当前仓库现状：

1. `api/db.py` 里已经有 `EvaluationJob`
2. `evaluate_rag.py / evaluate_rag_answers.py / run_delivery_checks.py` 现在都还是文件驱动的评估入口，没有另一条评估专用的持久化主链
3. `.runtime/evaluation_jobs.db` 更像历史遗留或本地过渡态，不适合作为新的第一承接层
4. 如果再加一层本地 SQLite，只会多出一次同步和一次回填，不会提供新的业务能力

所以 Phase D 的顺序应该是：

1. 先把 job 级摘要写进主库 `evaluation_jobs`
2. 再继续保留 report JSON / Markdown / trace 作为文件真相源
3. 等 job 级索引稳定后，再考虑是否把 case 级聚合单独拆表

### 10.5 Phase D 第一刀最小实施顺序

只针对这一刀，建议按下面顺序推进：

1. 先把 retrieval / answer-level 的 job 摘要字段对齐到 `EvaluationJob`
2. 再把 `report_json_path / report_markdown_path / trace_paths_json` 这类回链字段补齐
3. 然后只补聚合层字段，不碰逐 case 明细
4. 最后再决定是否需要额外的 case 级索引表

这样能保证评估结果先能查、能跳回文件、能按状态和类型过滤，而不会一开始就把报告正文重构成另一套数据库真相源。

### 10.6 Phase D 第一实现切片：CLI 评估也进入 `evaluation_jobs`

在 API 路由之外，本地 CLI 入口现在也已经对齐到同一条 job 索引主路径：

1. `scripts/evaluate_rag.py`
   - 运行前会 `acquire_evaluation_run(evaluation_type="retrieval")`
   - 完成后通过 `mark_completed()` 回写 `report_json_path / report_markdown_path / trace_paths`
2. `scripts/evaluate_rag_answers.py`
   - 运行前会 `acquire_evaluation_run(evaluation_type="answer")`
   - 完成后通过 `mark_completed()` 回写 `report_json_path / report_markdown_path`
3. 两条 CLI 都会把解析后的 `dataset_path` 与 `request_payload` 写入 `EvaluationJob`

这意味着 Phase D 第一刀现在不再只是“API 任务能进主库，CLI 还是纯文件产物”，而是已经收成：

1. job 级元数据统一进入主数据库 `evaluation_jobs`
2. report JSON / Markdown / trace 仍继续留文件做真相源
3. 评估历史列表、按类型筛选、回跳到文件路径的能力不再依赖“必须从 API 路由触发一次”
