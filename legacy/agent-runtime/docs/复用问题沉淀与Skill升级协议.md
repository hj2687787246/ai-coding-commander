# 复用问题沉淀与 Skill 升级协议

更新时间：2026-04-11

## 1. 文档定位

这份协议文档只解决一件事：

把外部 harness 里“问题复用、结果沉淀、流程升级”的方法，接到当前仓库已经存在的指挥官体系里。

它不是第二套指挥官手册，也不是新的平行骨架说明。

一句话说：

**当前仓库吸收的是方法，不复制 `plans / memory / reports / debt` 这套外部目录骨架。**

---

## 2. 对应关系

外部 harness 常见部件和当前仓库的对应关系如下：

1. `plans`
   - 对应当前仓库的：
     - `commander/state/当前任务卡.md`
     - `commander/outer/新窗口启动指令模板.md`
     - `commander/transport/prompts/execution_window_task_template.md`
   - 含义：
     - 当前任务状态、下一步最小动作、开窗协议，已经由现有任务卡和模板承担

2. `memory`
   - 对应当前仓库的：
     - `commander/core/任命.md`
     - `commander/core/主文档.md`
     - `commander/state/问题索引.md`
     - `commander/outer/新窗口交接说明.md`
   - 含义：
     - 稳定事实、高频问题和接班规则，已经有现成文档体系承接

3. `reports`
   - 对应当前仓库的：
     - `commander/transport/prompts/execution_window_report_template.md`
     - 执行窗口固定汇报顺序
   - 含义：
     - 结果回收靠结构化汇报和验证证据，不靠聊天印象

4. `debt`
   - 对应当前仓库的：
     - `commander/state/问题索引.md` 里的开放问题
     - `commander/state/当前任务卡.md` 的活跃任务与阻塞项
     - 相关专题文档里的“风险 / 未完成项”
   - 含义：
     - 当前仓库已有记录风险和待跟进问题的落点，不需要再复制一层 debt 骨架

5. skill / reusable tool
   - 对应当前仓库的：
     - 本地 skill
     - `scripts/` 下可执行脚本
     - 现有模板与协议文档
   - 含义：
     - skill 负责复用流程，脚本负责可执行动作，仓库 Markdown 负责项目事实

---

## 3. 为什么不能复制平行骨架

当前仓库不能直接复制外部 `plans / memory / reports / debt` 平行骨架，原因有 4 个：

1. 当前仓库已经有指挥官文档体系
   - 再复制一套目录，只会让同一事实出现两份真相源

2. 当前仓库更强调“按文档接班”
   - 新窗口恢复上下文时，已经明确依赖现有 Markdown 文档，而不是仓库外平行记忆层

3. 当前任务回收协议已经存在
   - 开窗模板、任务模板、结果模板和任务卡，已经覆盖计划、执行、汇报、回收这条链路

4. 当前目标是稳定恢复真相源，不是再造框架
   - 第一优先级是“窗口压缩上下文后还能稳定回到文档真相源”
   - 不是把仓库改成另一个 harness 的目录外形

一句话规则：

**方法可以吸收，真相源不能分叉。**

---

## 4. 窗口恢复真相源的标准顺序

无论是新窗口、压缩上下文后的窗口，还是要处理重复问题的窗口，默认都按下面顺序恢复上下文：

1. 先读对应角色的 Markdown 启动文档
   - 指挥官窗口先读 `commander/core/任命.md`
   - 执行窗口先读 `README.md`、`commander/outer/agent_workbench.md` 和本轮指定文档

2. 再按主题懒加载相关 Markdown 文档
   - 例如任务卡、问题索引、专题说明、交接说明

3. 再判断当前问题应该落到哪里
   - 文档
   - 脚本
   - skill

4. 只有在需要验证或实现时，才继续读脚本或代码

5. 完成后先验证，再汇报，再决定是否沉淀

禁止的顺序是：

1. 先沿用聊天印象
2. 先写 skill
3. 再回头猜文档里原本怎么说

一句话规则：

**先回到仓库 Markdown 真相源，再决定复用载体。**

## 5. 各类资产的适用场景

### 5.1 任务卡

适用场景：

- 当前仍在推进的活跃任务
- 下一步最小动作
- 阻塞点和待决策项

不适用：

- 长篇背景
- 已经收口的稳定规则

### 5.2 问题索引

适用场景：

- 已经反复出现的高频问题
- 仍需要后续跟进的开放问题

不适用：

- 当前任务状态
- 具体执行步骤

### 5.3 主文档 / 任命文档 / 交接说明

适用场景：

- 稳定事实
- 角色边界
- 接班规则
- 懒加载地图

不适用：

- 具体一次执行窗口的临时实现细节

### 5.4 脚本

适用场景：

- 需要稳定重复执行的命令或检查
- 手工重复成本高、且容易出错的动作

不适用：

- 只在单次讨论里出现的判断规则
- 项目背景事实说明

### 5.5 skill

适用场景：

- 触发条件稳定
- 判断流程可复用
- 需要提醒窗口先读哪些文档、再做什么分流

不适用：

- 项目事实本身
- 随时间频繁变化的任务状态
- 一次性讨论结论

## 6. 什么问题留在文档层，什么问题升级成 skill

优先继续留在文档层的问题：

1. 主要价值是保存项目事实
2. 经常因为项目现状变化而更新
3. 需要和任务状态、时间线、专题文档紧密联动
4. 每次都必须先看仓库内最新内容才能回答

适合升级成 skill 的问题：

1. 重复出现
2. 触发条件稳定
3. 处理流程相对固定
4. 可复用的是“怎么读、怎么分流、怎么验证”
5. 边界清楚，不会偷偷替代项目文档本身

判断口诀：

**事实留文档，动作进脚本，流程进 skill。**

---

## 7. Skill 的最小升级判据

只有同时满足下面条件，才值得从“重复问题”升级成 skill：

1. 重复出现
   - 不是一次性的偶发问题

2. 触发条件稳定
   - 能明确知道什么时候该触发，什么时候不该触发

3. 流程可复用
   - 至少能复用“先读什么、再判断什么、最后怎么落点”

4. 有验证证据
   - 已经在真实窗口使用过，证明这种流程确实能稳定收口

5. 边界清楚
   - 明确知道 skill 不负责保存什么，不替代什么

如果只满足“经常被问”，但流程还不稳定，优先先写文档，不急着做 skill。

---

## 8. Skill 的轻量化规则

当前仓库里的 skill 必须保持轻量，遵守下面规则：

1. skill 只固化流程壳
   - 例如先读哪些 Markdown、怎样判断落到文档 / 脚本 / skill

2. skill 不复制项目事实
   - 不把 `指挥官文档.md`、任务卡、问题索引里的事实重新抄一遍

3. skill 必须回指仓库 Markdown 真相源
   - 需要知道项目事实时，先读文档

4. skill 不承担当前任务状态
   - 当前推进到哪里，仍以任务卡和相关文档为准

5. skill 不制造第二套记忆层
   - 它是流程入口，不是 memory 副本

---

## 9. 推荐沉淀顺序

当前仓库处理重复问题时，默认按下面顺序升级：

1. 先在文档里形成稳定表达
2. 如果需要稳定执行，再补脚本
3. 如果问题持续反复、且流程已经稳定，再补 skill

不要倒过来做成：

1. 先写 skill
2. 再补文档真相源

因为这样最容易让 skill 变成过时事实副本。

---

## 10. 本轮落地边界

这份协议落地后的直接用途是：

1. 给指挥官窗口一个统一口径
2. 给新窗口模板增加“skill / 复用工具沉淀”入口
3. 给后续本地 skill 提供轻量边界

本轮不做的事：

1. 不重写现有指挥官体系
2. 不新增 `plans / memory / reports / debt` 目录
3. 不把所有沉淀机制一次性改成 skill-first
4. 不改业务代码
