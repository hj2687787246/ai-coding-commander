# 指挥官运行时归档与清理规则

更新时间：2026-04-12

这份文档只定义 `.runtime/commander` 的归档与清理边界，不替代通用 `.runtime` 清理规则。
目标是让后续窗口一眼知道：哪些对象属于 live runtime，什么时候允许移出，移出时必须一起搬走哪些关联产物。

## 1. 分层边界

`.runtime/commander` 现在分成两层：

1. live runtime
   - `tasks/`
   - `improvements/`
   - `improvement_actions/`
   - `workers/`
2. archive runtime
   - `archive/tasks/`
   - `archive/improvements/`
   - `archive/improvement_actions/`

live runtime 负责当前控制面继续运行。
archive runtime 负责保留已归档任务的历史证据，不再参与当前调度。

## 2. 默认保留期

当前归档任务不是一归档就立刻移走。

默认规则：

1. 只有 `lifecycle_status = archived` 的 task 才允许进入 cleanup 判断。
2. 默认保留期是 7 天。
3. 只有超过保留期后，task 才会进入 `cleanup_eligible = true`。

这条规则由 `commander_harness.py` 的 `is_cleanup_eligible()` 统一判断。

## 3. 归档搬迁对象

当 `commander_archive_cleanup.py` 实际搬迁一个 cleanup-eligible archived task 时，必须一起搬走：

1. `tasks/<task_id>/`
2. `improvements/<task_id>.candidate.json`（如果存在）
3. `improvement_actions/<candidate_id>/` 下所有 `candidate_metadata.json.task_id = <task_id>` 的动作目录

也就是说，task、candidate、apply 产物现在按同一个 task 维度一起离开 live runtime，避免出现“任务已归档，动作产物还残留在 live runtime”的半截状态。

## 4. 安全边界

`commander_archive_cleanup.py` 默认是 dry-run。

当前安全规则：

1. 非 `archived` task 不移动。
2. 未达到保留期的 archived task 不移动。
3. 如果 archive 目标路径已存在，任务不会强行覆盖，而是返回显式原因。
4. 归档动作会写 `archive_manifest.json`，记录源路径、目标路径、关联 candidate、关联 improvement_actions 和 retention policy。

当前这条归档搬迁只负责文件层迁移，不负责删除数据库里的 commander task catalog 记录。

## 5. 正式入口

建议先看 dry-run，再决定是否 apply：

```powershell
.\.venv\Scripts\python.exe -m commander.transport.scripts.commander_archive_cleanup --runtime-root .runtime/commander --dry-run
.\.venv\Scripts\python.exe -m commander.transport.scripts.commander_archive_cleanup --runtime-root .runtime/commander
```

如果只想处理单个 task，可以再加：

```powershell
--task-id <task_id>
```

当前还新增了三条正式入口：

```powershell
.\.venv\Scripts\python.exe -m commander.transport.scripts.commander_archive_catalog --runtime-root .runtime/commander --sync
.\.venv\Scripts\python.exe -m commander.transport.scripts.commander_cleanup_plan --runtime-root .runtime/commander
.\.venv\Scripts\python.exe -m commander.transport.scripts.commander_maintenance --runtime-root .runtime/commander
```

- `commander_archive_catalog.py --sync`
  - 写出 `archive/catalog.json`
  - 同步 `archive/snapshots/*.summary.json`
  - 自动清理 stale snapshot
- `commander_cleanup_plan.py`
  - 默认 dry-run，显式给出 commander 专用保护名单
  - 盘点 live runtime 里已进入 `cleanup_eligible` 的 archived task
  - 检查 archive catalog snapshot 是否缺失、漂移或需要同步
  - `--apply` 时会执行 archive cleanup，并在收尾时自动刷新 archive catalog snapshot
- `commander_maintenance.py`
  - 默认执行单次维护周期：archive catalog sync、cleanup plan、audit
  - `--apply-cleanup` 时会在同一周期中执行 cleanup-eligible archived task 搬迁
  - 输出 `cycle_health` 与 `recommended_actions`，用于判断本轮维护是 `healthy`、`cleanup_pending` 还是 `needs_attention`
  - `--skip-archive-sync` 可用于只做 cleanup/audit 视角的维护烟测

## 6. 当前范围外

这轮规则暂时还不处理：

1. 数据库 catalog 归档记录的收缩或删除
2. `workers/` 目录的长期历史压缩
3. 更重的 archive compaction / secondary storage 策略
4. 应用外部的定时调度安装，例如系统级计划任务或 Codex automation

如果后续要继续推进，优先方向是：

1. 如果 archive runtime 真正变大，再补更重的 archive compaction / secondary storage
2. 如果需要无人值守运行，再单独接入外部定时调度；仓库内当前稳定入口是 `commander_maintenance.py`
3. 数据库 catalog 的 archive 生命周期策略
