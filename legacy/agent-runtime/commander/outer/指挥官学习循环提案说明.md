# 指挥官学习循环提案说明

更新时间：2026-04-11

## 1. 文档定位

这份说明只解释一件事：

任务回收后，怎样把“这次是不是值得沉淀”压成结构化候选，而不是继续散在聊天里。

一句话说：

**先产出 candidate，再决定吸收，不直接自动生效。**

---

## 2. 当前入口

当前入口是：

```powershell
.\.venv\Scripts\python.exe scripts\commander_propose_improvement.py --report <report.json>
```

默认输出会写到：

```text
.runtime/commander/improvements/
```

---

## 3. 候选会分流到哪里

当前 v1 只分 4 类：

1. `doc`
2. `script`
3. `skill`
4. `none`

含义是：

1. `doc`
   - 更适合沉淀成稳定项目事实、解释或入口说明
2. `script`
   - 更适合沉淀成可重复执行动作
3. `skill`
   - 更适合沉淀成可复用流程壳
4. `none`
   - 暂时不建议沉淀，继续观察

---

## 4. 当前判断依据

v1 的判断依据保持轻量：

1. `report.json` 的 `changed_files`
2. `summary`
3. `verification`
4. `recommended_next_step`

它当前不是智能规划器，也不是复杂裁判模型。

它只是把已有回收结果压成一个稳定 candidate，方便后续判断：

1. 留在文档
2. 升级脚本
3. 升级 skill

---

## 5. 当前边界

当前 v1 只做：

1. 读取已存在的 report
2. 推断推荐落点
3. 生成结构化 candidate

当前 v1 不做：

1. 不自动写回文档
2. 不自动改脚本
3. 不自动改 live skill
4. 不替代指挥官最终判断

所以它更像是：

**受控学习循环里的“提案层”，不是自动执行层。**
