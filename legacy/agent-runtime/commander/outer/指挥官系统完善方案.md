# 指挥官系统完善方案

更新时间：2026-04-12

## 1. 目标定位

当前指挥官系统已经具备可运行的最小闭环，但还不是一套完整、可长期维护的控制面。

这条主线的目标不是继续按 `harness v1` 思路补微型缺口，而是把它完善成一套：

1. 可持续调度
2. 可持续回收
3. 可持续归档
4. 可持续恢复
5. 可持续学习
6. 可持续治理

的一体化指挥官系统。

一句话定义：

**把当前“文档 + transport + runtime 落点”的轻量闭环，升级成“有生命周期治理、有审批链、有对账能力、有清理规则”的受控控制面。**

---

## 2. 完成标准

当下面 6 组能力全部成立时，才算“完善系统”，而不是“补了几个点”。

### 2.1 任务生命周期闭环完整

任务在 runtime 层必须有完整生命周期，而不是长期停在 `awaiting_report` 或 `ready_to_close`：

1. `intake`
2. `dispatched`
3. `running`
4. `report_ready`
5. `ingested`
6. `ready_to_close`
7. `archived`
8. `cleanup_eligible`

同时保留分支态：

1. `blocked`
2. `pending_user`
3. `canceled`
4. `stale`

### 2.2 文档态与 runtime 态一致

以下几层不能再各说各话：

1. `commander/state/当前任务卡.md`
2. `.runtime/commander/tasks/*/status.json`
3. `.runtime/commander/tasks/*/checkpoint.json`
4. commander task catalog

用户看到“已归档”，runtime 里就不能还挂着待关闭任务。

### 2.3 学习链闭环完整

当前只有：

`report -> improvement candidate`

完善后需要补成：

`report -> candidate -> review -> approved/rejected -> applied -> archived`

并区分 3 类吸收目标：

1. doc
2. script
3. skill

### 2.4 worker 治理完整

warm worker 池不能只停留在文件型 acquire/release。
完善后需要有：

1. lease / heartbeat
2. stale worker 检测
3. orphan task 检测
4. reuse 边界
5. close / recycle 规则
6. 并发安全

### 2.5 观测与对账完整

指挥官不能只靠单个 `status.json` 看当前状态，还要能做：

1. 全量任务总览
2. 悬空任务扫描
3. 文档态 / runtime 态 / catalog 态对账
4. 失败显式暴露，而不是静默吞掉
5. 恢复入口统一

### 2.6 运行时卫生完整

`.runtime/commander` 不能无限积累历史任务和快照。
完善后需要明确：

1. 保留期
2. 归档期
3. 可清理期
4. 清理白名单 / 黑名单
5. 安全 dry-run
6. 自动或半自动清理入口

## 3. 当前系统距离“完善”还差什么

当前已经具备：

1. `dispatch -> worker_report -> ingest -> report/status/checkpoint`
2. trusted ingest 后自动产出 improvement candidate
3. resume / status 恢复锚点
4. warm worker 池最小能力
5. commander task catalog 最小汇总能力

当前仍然缺：

1. runtime 终态归档动作
2. 悬空任务治理
3. candidate 审批与吸收状态机
4. worker lease / stale / orphan 治理
5. catalog 与 runtime 的强一致或至少显式不一致提示
6. commander runtime 自身的归档与清理策略

所以当前系统应理解为：

**最小闭环已成立，但系统治理闭环尚未成立。**

---

## 4. 完善工程的 6 条主线

### A. 生命周期与收口治理

目标：
让任务从派发到归档，在 runtime 层也真正结束。

需要补齐：

1. runtime 终态定义
2. `close / archive / cancel / reopen` 动作
3. `stale` 与超时规则
4. `当前任务卡` 与 runtime 的同步规则
5. 任务收口后的自动移出 / 半自动移出策略

建议脚本入口：

1. `commander_close.py`
2. `commander_archive.py`
3. `commander_reconcile.py`

### B. 学习链审批与吸收治理

目标：
让 improvement candidate 不再只是候选文件，而是进入正式审批链。

需要补齐：

1. candidate 状态机
2. review / approve / reject 动作
3. doc/script/skill 三类吸收入口
4. applied 后的归档与回链
5. 避免重复候选

建议状态：

1. `candidate`
2. `approved`
3. `rejected`
4. `applied`
5. `archived`

### C. 调度与执行治理

目标：
让“为什么继续、为什么暂停、为什么回用户、为什么拆单”都进入显式控制逻辑。

需要补齐：

1. task owner / source / parent-child 关系
2. split / handoff / reopen 规则
3. user decision gate 与 commander decision gate 的统一定义
4. pending-close attention 的正式消费动作
5. 子 agent 结果回收后的强制分级与下一步约束
### D. Worker Pool 与并发安全

目标：
让 worker 池从“可复用”升级成“可治理”。

需要补齐：

1. lease
2. heartbeat
3. stale worker reclaim
4. orphan worker reclaim
5. worker 与 task 的绑定对账
6. 文件锁或数据库锁
7. 同一写集合的 single-owner 规则

并发治理补充硬规则：

1. 并行是允许的，但只能发生在不重叠写集合之间。
2. 同一实现切片如果已经派给某个 worker，指挥官本地或其他 worker 不得再对同一写集合并行施工。
3. `wait_agent` 超时只代表“暂时未返回”，不代表可以默认开第二个执行者去写同一块。
4. 如果确实需要改派，必须先发生显式 ownership 切换，再开始新执行者的施工。

### E. Catalog 与观测面

目标：
让指挥官能在总览层看出系统是否健康，而不是逐个点开文件。

需要补齐：

1. catalog 同步失败显式告警
2. runtime / catalog 差异扫描
3. stale tasks 视图
4. pending close 视图
5. pending candidate review 视图
6. 最近可信完成项总览

### F. Runtime 卫生与保留策略

目标：
让 `.runtime/commander` 可长期运行，不被旧任务与快照拖垮。

需要补齐：

1. reports/events/checkpoints 的保留期
2. archived task 的转存规则
3. cleanup eligibility 判断
4. commander 专用清理白名单
5. 清理前对账
6. 定期清理或手动清理入口

---

## 5. 推荐推进顺序

不要六条线一起开。
推荐按下面顺序推进：

1. A 生命周期与收口治理
2. B 学习链审批与吸收治理
3. C 调度与执行治理
4. D Worker Pool 与并发安全
5. E Catalog 与观测面
6. F Runtime 卫生与保留策略

原因很简单：

1. 不先解决生命周期，系统永远会有悬空任务
2. 不先解决审批链，学习闭环就会卡在 candidate
3. 不先解决治理语义，后面的 worker 和观测都只是表面增强

---

## 7. 非目标

这条完善主线当前不追求：

1. 把项目改造成通用多 agent orchestration 平台
2. 让系统无审批地自动改 live skill
3. 在没有治理闭环前继续扩张更多花哨能力

当前优先级不是“更炫”，而是“更稳、更可治理、更能长期运行”。
