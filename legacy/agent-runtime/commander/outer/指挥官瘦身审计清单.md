# 指挥官瘦身审计清单

更新时间：2026-04-14

## 1. 定位

这份清单只回答一个问题：

**哪些东西属于 AI coding harness 的核心，哪些只是 Agent 仓库实验出来的重运行时适配层。**

本轮不删除代码。先把边界写清楚，后续如果要删，必须按“依赖迁移 -> 回归验证 -> 删除”的顺序做。

一句话结论：

**保留驾驶舱，冻结平台化实验；先不把 LangGraph / host runtime / worker pool 当成通用指挥官 skill 的必要部分。**

## 5. 不建议做

1. 不建议为了“清爽”立刻删除 `commander/graph` 整包；目前治理核心仍有路径级耦合。
2. 不建议继续新增 host daemon、session mailbox、warm worker 复用功能；这些更像平台化 runtime。
3. 不建议把 `superpowers` 的 TDD / plan / review 流程重写进本仓库；应优先适配和复用。
4. 不建议把 Agent 仓库实验层同步到其它仓库；其它仓库默认只用通用 `commander-mode` skill，并按需建立自己的项目 skill / references。
5. 不建议继续把 spec、plan、工作模式和 harness 边界扩写进 `AGENTS.md`。

---

## 6. 下一刀建议

如果继续瘦身，优先做“无删除迁移”：

1. 新建或选定中性路径承载核心治理：例如 `commander/transport/policies/`。
2. 先迁移 `role_guard.py`、`lane_contract.py`、`tool_path_governance.py`、provider registry 的最小依赖。
3. 让 `commander_dispatch.py`、`commander_stop_gate.py`、`commander_audit.py` 不再 import `commander.graph.*`。
4. 迁移完成并验证后，再把 `commander/graph` 标成可删除实验层。

这样做的目标不是继续造框架，而是把已经有用的 harness 核心从重 runtime 路径里剥出来。
