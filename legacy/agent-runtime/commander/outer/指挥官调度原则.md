# 指挥官调度原则

更新时间：2026-04-11

这份文档只回答一个问题：

**当前指挥官系统为什么需要 harness 的“手脚”，以及这层手脚应该按什么原则实现。**

它不是第二份指挥官手册，也不是实现细节说明。

它当前的边界也要明确：

- 这是指挥官系统的外圈原则文档
- 不是指挥官系统本体真相源
- 如果与 `指挥官任命.md`、`指挥官文档.md` 冲突，始终以前两者为准

---

## 1. 先给结论

当前系统已经有：

- 脑子：指挥官负责判断、拆任务、控边界、回收结果
- 规矩：文档真相源、开窗模板、汇报模板、任务卡、时间线

当前系统还缺：

- 手脚：把任务稳定发给执行窗口、把结果稳定收回指挥官的 transport 层

一句话说：

**我们现在不是缺更多提示词，而是缺一层最小调度与回收 harness。**

---

## 2. 核心链路

当前系统目标链路必须固定为：

`用户 -> 指挥官 -> 执行载体 -> 指挥官 -> 用户`

含义：

1. 用户只对指挥官下达目标、约束和决策
2. 指挥官决定是否留在当前窗口收口，还是派给执行载体
3. 执行载体只负责实现、验证、汇报
4. 执行载体遇到阻塞、分叉、缺信息时，先回指挥官
5. 最终只有指挥官统一对用户汇报

因此：

- 执行载体不是直接面向用户的长期对话终端
- 用户也不应该承担“在多个执行窗口之间搬运上下文”的 transport 角色

当前默认执行载体优先级是：

1. 子执行 agent
2. 明确隔离时再开外部执行窗口

---

## 3. 为什么不能继续靠长提示词串联

如果单步成功率是 95%，连续 20 步的理想串联成功率大约只有：

`0.95^20 ≈ 35.8%`

所以问题不只是：

- 提示词再润色一点
- 每个窗口再聪明一点

真正的问题是：

- 长链自然语言 handoff 会累计误差
- 单点准确率高，不能掩盖端到端可靠性低

因此当前系统要做的不是继续堆更长提示词，而是：

1. 减少纯自然语言串联步数
2. 把关键 handoff 变成结构化合同
3. 在关键节点回到指挥官，而不是让执行窗口自由长跑
4. 用脚本和状态机替代一部分 prompt 理解

---

## 4. 手脚层的设计原则

### 4.1 真相源不分叉

- 项目事实继续留在仓库 Markdown 文档
- 不复制 `plans / memory / reports / debt` 平行骨架
- harness 只负责调度和回收，不制造第二套长期记忆

### 4.2 结构化优先于长文本

- 指挥官发任务时，优先产出 machine-readable task packet
- 执行窗口回结果时，优先产出 machine-readable task report
- 自然语言只保留给人读的解释层，不承担唯一真相源职责

### 4.3 执行窗口只做 worker

- worker 的职责是：读任务、执行、验证、汇报
- worker 不负责和用户直接谈判
- worker 不负责自己扩张成新项目经理

### 4.4 阻塞必须回指挥官

执行窗口遇到下面情况时，默认不继续长跑，而是回指挥官：

1. 缺关键信息
2. 出现方案分叉
3. 验证失败且无法在当前边界内收口
4. 任务边界变化，可能需要新开窗口

### 4.5 少步、短链、强检查点

真正要提高端到端成功率，不是把每一步从 95% 提到 97%，而是：

1. 把 20 个自然语言节点压到 4 到 6 个关键检查点
2. 每个检查点都能验证
3. 每个检查点都能被指挥官接管

---

## 5. harness v1 应该做到什么

最小 v1 只需要解决 3 件事：

1. dispatch
   - 指挥官把任务压成结构化 packet
2. ingest
   - 指挥官稳定回收 worker 的结构化结果
3. status
   - 当前 task / report / event 状态可追踪

最小 v1 不做：

1. 不做大而全多 agent 框架
2. 不做自动决策替代指挥官
3. 不做第二套长期记忆系统
4. 不强行自动开窗到所有场景

---

## 6. v1 的推荐边界

建议最小落点：

- `commander/transport/schemas/commander_task_packet.schema.json`
- `commander/transport/schemas/commander_task_report.schema.json`
- `commander/transport/scripts/commander_dispatch.py`
- `commander/transport/scripts/commander_ingest.py`
- `commander/transport/scripts/commander_status.py`
- runtime 目录，例如 `.runtime/commander/`

建议最小状态：

- `done`
- `blocked`
- `need_split`

指挥官回收后再统一判断：

- 已收口
- 部分完成
- 阻塞待决策

---

## 7. 和当前仓库的分工

- `commander/core/任命.md`
  - 负责身份、硬规则、懒加载地图
- `commander/core/主文档.md`
  - 负责稳定事实、职责边界、核心交互模型
- `commander/state/当前任务卡.md`
  - 负责当前活跃任务和下一步最小动作
- `commander/state/时间线.md`
  - 负责收口后的完成节点
- `prompts/*`
  - 负责当前人工派工与汇报模板
- harness v1
  - 负责让派工和回收逐步脱离人工复制粘贴

一句话说：

**文档是真相源，指挥官是控制面，harness 是 transport，执行窗口是 worker。**

---

## 8. 当前最重要的一条判断

如果后续实现让用户继续承担这些工作：

- 手工复制任务给执行窗口
- 手工搬运执行结果回指挥官
- 手工在多个窗口之间同步状态

那它还不算真正补上了 harness 的手脚。

只有当这些动作逐步被结构化 task packet、task report、dispatch、ingest 替代时，当前系统才算从“文档化指挥协议”继续推进到“可执行调度系统”。
