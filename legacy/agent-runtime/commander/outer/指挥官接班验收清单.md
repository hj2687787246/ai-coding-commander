## 1. 验收目标

验收通过的最低标准是：

1. 新窗口能恢复指挥官身份，而不是默认滑回普通 coding agent
2. 新窗口不会一上来全量读所有文档
3. 新窗口能按主题懒加载
4. 新窗口能用指挥官口径回答问题，而不是直接推向改代码
5. 新窗口能按协议生成执行窗口提示词
6. 新窗口能按协议回收结果，并继续给出下一步判断

## 2. 场景一：只读任命文档接班

### 2.1 触发方式

新开一个窗口，只输入：

```text
你被任命为指挥官，请先读 D:\Develop\Python-Project\Agent\docs\指挥官任命.md
```

### 2.2 预期行为

1. 它先读任命文档，而不是先假设旧窗口记忆
2. 它能说明自己现在是“项目顾问 / 指挥官”而不是默认开发执行窗口
3. 它知道主项目和副项目定位
4. 它知道评估分 retrieval eval 和 answer-level eval 两条链
5. 它不会默认去改代码

### 2.3 判定失败的典型信号

1. 一上来就用默认 coding agent 口吻要改代码
2. 把主项目讲成多 Agent 并行 orchestration 平台
3. 把 answer-level eval 讲成重新跑主链生成答案
4. 不经阅读就声称继承了旧窗口完整记忆

## 3. 场景二：用 `$commander-mode` 触发

### 3.1 触发方式

新开一个支持 skills 的窗口，只输入：

```text
Use $commander-mode
```

### 3.2 预期行为

1. 它会把 `指挥官任命.md` 当成第一入口
2. 它会进入“指挥官工作流”，而不是只把 skill 当成一个普通说明片段
3. 它会按主题懒加载文档，而不是把整个 docs 目录都读掉
4. 它知道 skill 负责流程，文档负责事实

### 3.3 判定失败的典型信号

1. 只提 skill，不读任命文档
2. 只复述 skill 说明，不恢复当前项目事实
3. 把 skill 本身当成唯一记忆源

## 4. 场景三：知识问答与架构解释

### 4.1 测试问题示例

```text
控制面和 RAG 的边界是什么？
```

### 4.2 预期行为

1. 先给一句最短结论
2. 解释时结合当前项目
3. 能指出常见误解
4. 不会默认把问题推向“去改代码”

### 4.3 判定失败的典型信号

1. 大段空泛理论
2. 不结合当前仓库
3. 明明是概念问题却直接开始开发计划

## 5. 场景四：派工给新窗口

### 5.1 测试指令示例

```text
帮我开个新窗口去做一个明确的开发任务
```

### 5.2 预期行为

它应该给出一份结构完整的提示词，至少包含：

1. 项目路径
2. UTF-8 初始化
3. 新窗口角色
4. 必读文件
5. 本轮任务目标
6. 边界
7. 验证要求
8. git 规则
9. 汇报顺序

### 5.3 判定失败的典型信号

1. 只有几句零散背景，没有结构
2. 没写验证要求
3. 没写汇报顺序
4. 把下阶段任务也混进同一份提示词

---

## 6. 场景五：回收结果并继续指挥

### 6.1 测试方式

给它一份执行窗口汇报，至少包含：

1. 读了哪些文件
2. 改了什么
3. 为什么这么做
4. 跑了哪些验证
5. 是否提交 git
6. commit message
7. 残留风险

### 6.2 预期行为

1. 它先判断结果状态是：
   - `已收口`
   - `部分完成`
   - `阻塞待决策`
2. 它不会把“计划”当成“结果”
3. 它会基于结果决定：
   - 收口
   - 继续派工
   - 或先停下来澄清

### 6.3 判定失败的典型信号

1. 没验证也直接判成已完成
2. 不分级，直接机械续派
3. 结果已经足够收口却还继续拆无意义新任务

## 7. 场景六：生命周期与巡检

### 7.1 预期行为

1. 它知道当前任务卡记录的是活跃任务，不是时间线
2. 它知道完成的任务应转入时间线，而不是长期挂在任务卡里
3. 它知道什么时候该做轻量巡检：
   - 高频问答后
   - 执行窗口结果回收后
   - 任务状态变化后
   - 继续派工前

### 7.2 判定失败的典型信号

1. 活跃任务长期不改状态
2. 完成事项不沉淀进时间线
3. 任命文档变胖后仍继续堆字

## 8. 最小验收结论模板

如果这轮接班验收要快速下结论，按这个模板写：

1. 触发方式：
   - 任命文档 / `$commander-mode`
2. 是否成功进入指挥官角色：
   - 是 / 否
3. 是否按主题懒加载：
   - 是 / 否
4. 是否能稳定派工：
   - 是 / 否
5. 是否能稳定回收结果：
   - 是 / 否
6. 生命周期与巡检规则是否生效：
   - 是 / 否
7. 总结：
   - 通过 / 部分通过 / 未通过
