## 文档目的
这份文档用于在新窗口中恢复当前窗口承担的职责、规则和最小必要上下文。

它不是完整聊天记录，也不是阶段流水账，而是后续窗口真正需要的工作事实。

它当前的边界也要明确：

- 这是指挥官系统的外圈交接入口
- 不是指挥官系统本体
- 如果与 `指挥官任命.md`、`指挥官文档.md` 冲突，始终以前两者为准

## 关键工作方式
- 多窗口之间不共享运行时记忆。
- 新窗口不能假设知道旧窗口上下文。
- 长上下文容易导致幻觉、串线和旧结论污染当前任务。
- 当前采用的方法是：显式文档交接 + 新窗口读取最小必要上下文后执行。
- 这是当前阶段的工程妥协方案，不是理想终态。

## 当前最小共识
- 主项目更偏知识型 RAG、Agent-first、可观测性和会话恢复治理。
- 副项目更偏 Structured Output、Tool Calling、Workflow Automation 的 MVP 验证。
- 复杂 query 在主项目里不是通用多意图并行执行，而是单主路由为主、受控混合拆分为辅。
- 当前 retrieval eval 和 answer-level eval 已分链：
  - retrieval eval 负责检索质量
  - answer-level eval 负责答案质量
- 如果 retrieval eval 开了 `with_traces`，语义是“评估借一次真实查询链路拿 execution trace，并优先复用同次真实执行产物”；不是“线上真实请求顺手替评估打工”。
- 真实请求本身现在也会顺手写轻量 `execution summary`；后续排查最近真实会话时，优先读 summary，再决定要不要补跑评估 trace。

只有在 compact 锚点不足以解释当前状态时，才再打开 `checkpoint.json`、任务卡或时间线。
