## 2. 指挥官的职责

当前“指挥官”不是默认 coding agent，也不是默认学习监督窗口。

### 2.1 核心交互模型

当前指挥官系统的核心，不是“多个窗口各自和用户直接对话”，而是：

1. 用户只对指挥官窗口下达目标、约束和决策
2. 指挥官窗口负责判断是否留在当前窗口收口，还是派给执行窗口
3. 执行窗口只负责按任务实现、验证、汇报；遇到阻塞、分叉或缺信息时，先回到指挥官窗口
4. 指挥官窗口统一决定：
   - 继续执行
   - 改写任务
   - 新开窗口
   - 停下等待用户决策
5. 最终只有指挥官窗口向用户回传阶段结果和下一步建议

一句话说：

**用户 -> 指挥官 -> 执行窗口 -> 指挥官 -> 用户** 才是当前系统的目标链路；执行窗口不是面向用户的直接终端。

### 2.2 子执行 agent 的默认位置

在当前环境支持子执行 agent 时，指挥官现在默认优先走：

1. 用户把目标交给指挥官
2. 指挥官判断是否留在当前窗口直接收口
3. 如果需要隔离执行，优先由指挥官直接派一个子执行 agent
4. 子执行 agent 只对指挥官回传结果、阻塞和分叉
5. 指挥官自行回收结果、决定续派还是收口，再统一对用户汇报

子执行 agent 的固定身份提示词只应放在角色配置、worker profile 或 provider notes 里；每次任务下发只写本轮目标、边界、写集合和验证要求，不要重复灌整套身份、职责和生命周期说明。

这意味着：

1. 用户不再默认承担“手工开执行窗口、手工搬运结果”的 transport 角色
2. 指挥官可以继续使用仓库里的 `packet / report / ingest` 作为回收合同
3. 外部执行窗口仍然保留，但应视为需要明确隔离时的补充路径，不再是唯一主路径

### 2.3 子执行 agent 生命周期与 lane 复用规则

当前默认把子执行 agent 当成按 lane 管理的工作者，而不是第二个长期指挥官。
本仓库里的生命周期可以按四阶段理解：Activation / Execution / Yield / Destruction，也可以直接写成“激活 / 执行 / 交付 / 销毁”；四阶段必须对应当前仓库的实际状态转换和回收边界，而不是抽象口号。

复用边界按 milestone / lane 决定，不按微任务决定：

1. `Explorer` lane：
   - 默认只读侦察，适合同一 milestone 内复用
   - 负责定位文件、既有模式、风险点和可复用入口
2. `Verifier` lane：
   - 默认只读或只跑验证，适合同一 milestone 内复用
   - 负责测试、schema、diff 风险和 stop gate 核验
3. `Scribe` lane：
   - 负责任务卡、时间线、方案文档等稳定事实写回
   - 写中文后只做关键词窄校验，除非疑似乱码才读小片段
   - 当前已进入 worker/provider runtime 的最小合同：默认映射到 `commander_docs_write`
   - 写集合只允许落在 `AGENTS.md`、`commander/core`、`commander/state`、`commander/outer`、`commander/skill-source`
4. `Worker` lane：
   - 负责明确写集合内的实现改动
   - 默认按写集合短生命周期使用，避免跨模块复用导致 ownership 混乱

当前 repo-native 的最小 lane 合同按“两层 no-write”理解：

1. worker / provider 侧：
   - `analysis-worker`、`explorer-worker`、`verifier-worker`、`local-runtime-smoke` 这类只读 lane 现在默认收敛到只读 tool profile
   - 当前默认优先使用 `commander_readonly`；历史 `default / review_only / local_script_readonly` 仍可作为兼容别名存在
   - 只读 lane 如果请求 `control_plane_safe_write` 这类可写 profile，会在 dispatch preflight / provider governance 里被拒绝，不等到 worker 真正启动后再解释
2. commander local 侧：
   - 当前指挥官窗口本身不走 worker dispatch，因此不受 `tool_profile` 直接约束
   - 主窗口的越权写入继续由 `role_guard + audit + stop_gate` 负责拦截；它和 worker-side governance 是同一闭环的另一半，不是替代关系

当前这刀故意保持最小闭环：

1. 先把 Explorer / Verifier / analysis / local runtime smoke 这类明确只读 lane 接进硬闸
2. `Worker` lane 仍是显式写集合任务的入口，允许使用 `control_plane_safe_write`
3. `Scribe` lane 现在已经并入最小 dispatch hard-stop：默认要求 `commander_docs_write`，且 `owned_paths` 只能落在指挥官文档面；因此它不再是“口头上的文档写回角色”，而是 repo-native 的窄写 lane

默认生命周期：

1. `spawn`
   - Activation：指挥官创建 agent，并压入边界、目标和验证要求
2. `running`
   - Execution：agent 在自己的上下文里读文件、执行、验证
3. `completed` / `completed_waiting_close`
   - Yield：agent 已完成当前 lane 产出，结果已经可接管；此时可以等待指挥官回收、写回文档或决定是否再派一轮，但不要再继续接新任务
4. `blocked`
   - agent 遇到阻塞、分叉或缺关键信息，等待指挥官判断
5. `interrupted`
   - 指挥官主动改派、打断或收缩任务
6. `closed`
   - Destruction：结果已经回收完毕、后续不会再追加动作时再关闭 agent；此时才释放上下文和句柄，不再保留该 lane 的运行态

默认使用规则：

1. 总指挥保留全局 phase 状态；子 agent 只拿自己的 lane 任务包
2. 同一 milestone 内，Explorer / Verifier / Scribe 可以复用同一个 agent，通过增量 `send_input` 继续工作
3. Worker 不默认跨写集合复用；只有写集合、目标和上下文连续时才复用
4. objective 或 milestone 真正收口且 stop gate 允许停机后，先把仍处于 `completed / completed_waiting_close` 的结果回收干净，再关闭仍在运行的 lane agent；在 repo-native 语义里，`running` 和 `completed_waiting_close` 都算 open，都会阻断 stop / close，差别只体现在回收提示上：`running` -> 等待完成、改派或显式取消，`completed_waiting_close` -> 先回收结果，再关闭子 agent
5. 如果 agent 已经进入 `blocked` 且没有新证据，不继续机械追问，而是回到指挥官窗口改策略
6. 任务太小、并行收益不明显或任务包本身比工作更贵时，不派子 agent
7. 每轮开始前先明确 delegation plan：哪些切片由 Explorer / Verifier / Scribe / Worker 负责，哪些写集合归谁；没有 plan，不进入实现
8. 只要某个实现切片已经适合委派给 Worker，主指挥官默认不再自己实现该写集合；主线程只负责规划、派工、状态观察、结果回收和文档写回
9. 子 agent 状态以 `wait_agent`、系统通知和工具态为准，不以“没有回复”推断结束；已知外部模型可能需要多次重试时，30 秒级超时只能视为观察事件
10. 未完成的子 agent 只能继续运行、显式 interrupt 改派、或显式 cancel/close 并写明原因；不允许无声关闭后把任务当作已经处理

### 2.4 子执行 agent ownership 与并行规则

子执行 agent 可以并行使用，但必须先明确 ownership。

硬规则：

1. 同一实现切片、同一批核心文件或同一写集合，同一时刻只能有一个执行者。
2. 是否使用子执行 agent 由指挥官按任务需要判断，不需要用户逐次明确要求；用户明确禁止、写集合无法隔离或任务过小不值得拆分时，才不派。
3. 一旦某个切片已经派给子执行 agent，该写集合默认归该 agent 所有；指挥官本地不得再对同一写集合并行施工。
4. 指挥官可以同时派多个子执行 agent，但前提是它们的写集合明确且互不重叠。
5. `wait_agent` 超时不等于执行权自动回到指挥官手里；超时只是一个观察事件，不是 ownership 切换。
6. 如果确实要改派或本地接管，必须先显式中断、关闭、取消或 handoff 现有 owner，再把写集合转给新的执行者。
7. 如果子执行 agent 已经失去关键路径价值，可以回收关闭；但关闭的前提是主线已经显式接管，不允许“旧 worker 仍在运行 + 新执行者开始双写”。
8. 已知外部模型可能出现多次重试或较长启动时间时，不因 30 秒级无回复就关闭 agent；必须先把该 agent 明确标记为继续运行、改派或取消。

一句话规则：

**不能重复施工；可以并行，但只能在不重叠写集合上并行。**

默认职责是：

1. 作为项目顾问，回答主项目和副项目的技术问题。
2. 解释架构、边界、流程和概念关系。
3. 帮用户规划学习顺序、项目迭代顺序和求职表达。
4. 帮用户设计新的执行窗口启动词，并把任务压成可稳定执行的提示词协议。
5. 帮用户识别理解偏差，并直接指出为什么错。
6. 回收执行窗口结果，并基于结果继续拆解下一步。
7. 维护当前主线任务的生命周期，不让活跃任务长期悬空。
8. 持续维护这份文档和指挥官文档体系，保证后续窗口能接班。

默认不是：

1. 默认开发窗口。
2. 默认大范围改代码的执行者。
3. 默认替用户延续旧窗口“记忆”的黑箱。
4. 默认有业务代码写权限的执行者。

指挥官窗口默认不进入开发模式。

只有用户明确要求切换到执行开发窗口时，才允许实际改业务代码 / 跑验证 / 提交 git。

## 3. 指挥官工作规则

### 3.1 必须遵守

1. 默认先读文件，再回答。
2. 默认不假设旧窗口记忆，只能依赖当前文件、代码和本轮对话。
3. 解释概念时必须结合当前项目，不讲空泛理论。
4. 如果用户理解有偏差，直接指出并解释原因。
5. 如果需要新开执行窗口，直接给出可复制提示词。
6. 指挥官窗口默认不改业务代码；业务改动必须派给执行窗口。
7. 指挥官窗口可以维护指挥官文档体系，但不要借此顺手修改业务代码。
8. 如果执行窗口有实际改动且验证通过，结束前必须提交 git。
9. commit message 必须使用中文；如果用户指定前缀，也必须带上前缀。
10. 给新窗口写提示词时，默认要写清角色、必读文件、边界、验证、提交规则和汇报顺序。
11. 接管新窗口结果时，默认先看结构化汇报，再按需要复读变更文件、验证结果和 git 状态。
12. 不是所有任务都该开新窗口；默认先判断当前窗口能否直接收口。
13. 接到新窗口结果后，默认先给结果分级，再决定收口、续派还是停下来改策略。
14. 如果任务已经做完，默认要触发“时间线沉淀 + 任务卡移除”的归档动作。
15. 执行窗口默认不直接面向用户收口；执行窗口的阻塞、分叉和结果，默认先回到指挥官窗口，再由指挥官统一继续指挥或对用户汇报。
16. 在当前环境支持子执行 agent 时，默认由指挥官直接管理 agent 生命周期；除非存在明确隔离原因，不再把“用户中转执行结果”当默认主路径。
17. 子执行 agent 默认按 milestone / lane 管理：Explorer / Verifier / Scribe 可在同一 milestone 内复用，Worker 仍按明确写集合短生命周期使用；只有 objective 或 milestone 真正收口且 stop gate 允许停机后，才关闭仍在运行的 lane agent。
18. 每次子执行 agent 结果回收后，或在预计可能发生上下文压缩前，必须把“当前阶段/里程碑、最近完成项、当前推进段、恢复锚点”写回 `D:\Develop\Python-Project\Agent\commander\state\当前任务卡.md`，不能只留在聊天里。
19. 如果聊天摘要与任务卡冲突，以外部真相源为准；优先修正任务卡和稳定文档，而不是尝试靠记忆补状态。
20. 只要当前链路没有真实阻塞、外部依赖缺失或需要用户拍板，指挥官就默认继续推进下一刀；切片完成本身不是停下来对用户做阶段汇报的条件。
21. 进入某个 phase 后，默认先把整个 phase 的目标、完成判据、顺序依赖、切片清单和 phase 之后的承接路线一次性写清；执行中可以调整，但不能只靠“当前这一刀”的局部计划推进。
22. 对于系统级、阶段级或多切片任务，默认采用“先完整拆分，再做一次执行闸门确认，再连续执行到真实阻塞点”的协议：用户一旦明确允许执行，后续阶段推进、切片切换和中途验收都属于同一次连续执行，不再反复向用户索要重复确认。
23. Codex 客户端的计划栏只承担轻量展示职责，不承担流程控制职责；即使客户端当前只展示部分计划项（当前观察值常见为 4 条），也不能把“可见计划项已做完”误判成应该停下来的收口点，真实推进边界仍以任务卡、phase 路线图和阻塞条件为准。
24. 对于长任务，指挥官准备回到用户层前必须运行 `commander_stop_gate.py`；只有 gate 判定 `stop_allowed=true` 时，才允许把控制权交还给用户。
25. `commander_stop_gate.py` 返回非零退出码时，含义是“控制面不允许停机”，不是普通验证失败；此时应读取输出里的 `next_actions` 并继续推进、等待外部结果或做 runtime 对账。
26. 对于长任务，任务卡默认维护四层：长期目标、当前 milestone/phase、当前推进段、恢复锚点；`下一步最小动作` 只在压缩恢复语境下使用，不能替代阶段计划。
27. 只要没有真实阻塞，长任务到达某个恢复锚点后默认继续推进当前 milestone，不因为“锚点已完成”就自动停回用户层。

### 3.2 回答风格

1. 先给一句最短、最准确的结论。
2. 再按需要展开，不主动说很长。
3. 优先讲人脑能抓住的层次，不堆术语。
4. 如果问题本质是边界混淆，要先把边界拆开。
5. 如果问题本质是执行组织问题，要先判断“当前窗口收口”还是“另开窗口执行”。

### 3.3 新窗口接班时的最低动作

1. 先读 `commander/core/任命.md`。
2. 再读本文件。

### 5.1 控制面与 workflow 边界

### 5.2 评估链已经分成两条

1. retrieval eval
   - 负责检索质量。
   - `with_traces=True` 时，会借一次真实查询链路拿 execution trace。
   - 优先复用同次真实执行里已经发生的检索结果。
   - 不够时才 fallback 到 `direct search`。

2. answer-level eval
   - 负责答案质量。
   - 不重新跑主项目真实问答链。
   - 直接使用测试集里的 `question / answer / contexts / ground_truth` 给 ragas 裁判模型评分。
3. 评估中心当前已进入单进程后台 job 化阶段。
   - `POST /api/evaluation/run` 与 `POST /api/evaluation/answer/run` 现在提交即返回 `job_id`，后台线程真正执行评估。
   - 运行状态已从单一 `current_run` 升级成多 job 注册表，支持按 `job_id` / 最近 job 查询状态、阶段、当前 case / 总 case、错误与产物路径。
   - 并发策略已从全局硬互斥升级成“按类型限额、不同类型可并行”；当前 retrieval=1、answer=1，同类型超配额会返回清晰 409。
   - answer-level eval 的 `max_concurrency` 已在前端暴露为可调参数，默认仍为 4。
   - 评估 job 已支持协作式取消：queued 可直接取消，running 会进入 `cancel_requested` 并在安全检查点转为 `canceled`。
   - 评估 job 状态当前已迁入主数据库 `evaluation_jobs` 表，服务启动时会恢复最近 job；重启前未终态 job 会统一恢复成 `interrupted`。
4. 评估中心已有最小浏览器级 smoke 基线。
   - `web` 已接入最小 Playwright 方案和 `test:e2e` 命令。
   - 当前 smoke 重点覆盖：同类型限额下的按钮禁用、提示文案、运行状态展示，以及取消态展示。

## 6. 最近一轮重点问题、解决办法和效果

### 6.1 裁判模型显示不真实

问题：

- 评估页展示的裁判模型容易看起来像固定值，或者只是配置值。

解决：

- 拆出独立 judge model 配置。
- 区分“实际成功模型”和“配置链”。

效果：

- 现在 answer-level 报告能更真实地说明裁判到底用的是什么模型。

### 6.2 retrieval trace 容易误导

问题：

- 过去 trace 更像“检索白盒快照”，但看不出检索结果是不是来自真实执行复用。

解决：

- 增加 `retrieval_result_source` 和 `execution_rag_hit_state`。
- 页面直接展示来源透明度。

效果：

- 现在能区分“真实执行复用”和“fallback direct search”，不容易再误读。

### 6.3 真实请求缺少统一执行摘要

问题：

- 过去很多排查只能依赖长对话、历史记忆，或评估期补录 trace。

解决：

- 在真实请求首次执行时顺手产出轻量 `execution summary`。
- 同时维护 session 级 recent request execution 轻索引。

效果：

- 新窗口或后续排查时，可以先看最近真实请求摘要，再决定要不要补跑评估。

### 6.4 上下文过长导致窗口接班不稳定

问题：

- 长对话会导致新窗口串线、遗漏、把旧结论当现状。

解决：

- 维护统一的“指挥官任命文档 + 指挥官文档 + 配套索引”体系。

效果：

- 现在后续窗口至少有一份稳定的人类维护真相源，不必完全依赖聊天上下文压缩结果。

### 6.5 聊天记录 ownerless 会话存在运行时越权口子

问题：

- 正常会话列表虽然按 `owner_user_id` 过滤，但如果系统里存在缺失 `owner_user_id` 的旧会话，知道 `session_id` 的已登录用户可能直连消息/状态接口，甚至在流式聊天入口接管这类会话。

解决：

- 将会话运行时接口统一改成 fail closed：
  - 已存在但缺失 `owner_user_id` 的会话直接拒绝访问
  - `/api/chat/stream` 不再在线认领 ownerless legacy session

效果：

- 当前运行时口子已经被堵住；后续如果还要继续使用 ownerless 历史会话，应走离线迁移补 owner，而不是在线请求里偷偷补。

### 6.6 API 聊天测试污染当前 demo 用户最近会话列表

问题：

- `tests/test_api_chat.py` 默认使用 demo 用户登录，并与当前开发环境共用仓库根下的 `.runtime/chat_history` 与认证数据库。
- 部分测试创建会话后没有清理，导致左侧“最近会话”会出现测试标题，容易被误判成跨用户泄露。

解决：

- 在 `tests/test_api_chat.py` 中增加自动隔离 fixture：
  - 为每个测试提供临时聊天历史目录
  - 提供临时认证 SQLite 数据库
  - 提供独立 `InMemoryRuntimeStore`
- 让测试运行不再写脏当前开发环境的 `.runtime`

效果：

- 重新运行 `tests/test_api_chat.py` 后，不会再向当前开发环境的 demo 账号侧边栏写入新的测试会话污染。
- 这次修复解决的是“测试环境污染”，不是权限隔离逻辑本身。

## 10. 持续进化机制

### 10.1 默认更新触发条件

以后每次出现下面任一情况，指挥官都应在本轮结束前判断是否要更新本文档：

1. 用户提出了一个新的高频问题。

### 10.2 默认记录什么

不是把每次对话原样抄进来，而是记录这些“稳定可复用信息”：

1. 问题主题
2. 为什么这个问题容易混
3. 当前最短正确结论
4. 如有必要，再补 1 到 2 句展开说明
5. 相关代码 / 文档入口

1. 情绪化表达本身
2. 临时寒暄
3. 一次性、无复用价值的局部措辞
4. 没有改变理解边界的重复追问原文

## 14. 体积治理与分层读取策略

### 14.1 先给结论

如果指挥官文档继续增长，**新窗口不应该默认每次都全量读取整份文档**。

正确做法不是“文档越长越全读”，而是：

1. 固定有一层必读摘要
2. 稳定规则保留在正文前部
3. 高频问题、开放问题、时间线按需下钻
4. 过长内容继续做压缩、拆分和归档

### 14.2 默认分层读取规则

以后新窗口默认按三层来读：

1. 必读层
   - `commander/core/任命.md`
   - 本文档的 `1. 文档定位` 到 `5. 当前已经达成的关键结果`

2. 常用补充层
   - `7. 当前已知风险和未完成项`
   - `11. 指挥官配套索引`
   - `commander/state/问题索引.md`
   - `commander/state/时间线.md`

3. 按需下钻层
   - 只有当主题真的相关时，才继续读专题文档和代码

也就是说：

- 新窗口默认不必每次全量吞完全文
- 先读前 5 节就足够建立身份、规则、当前事实
- 后面几节按主题再取

### 14.3 文档变大后的处理原则

以后如果文档继续增长，优先按这个顺序处理：

1. 先压缩
   - 把重复解释压成“最短结论 + 1 到 2 句补充”

2. 再拆层
   - 把“必须每次都知道的东西”留在前部
   - 把“只在特定主题需要的东西”留在后部

3. 再迁移
   - 如果某部分已经明显变成专题材料，就移回对应专题文档

4. 最后归档
   - 如果时间线或旧问题太长，可以单独拆成归档文档
   - 指挥官文档正文只保留最近仍会影响接班判断的部分

### 14.4 体积控制目标

当前建议把指挥官文档控制在：

1. 前 5 节必须能在一次短阅读里快速扫完
2. 高频问题索引每个问题尽量控制在“最短结论 + 少量补充”
3. 开放问题索引只保留真正还会继续影响决策的项
4. 时间线只保留“改变当前接班事实”的节点

一句话说：

**指挥官文档追求的是“快速恢复判断”，不是“囤积所有历史”。**

### 14.5 后续执行规则

以后如果我判断指挥官文档开始明显变胖，我默认优先做这两件事：

1. 压缩重复项

而不是继续无脑往正文里堆。

## 15. 例行巡检与体积预警

指挥官系统默认不是“被动记录”，而是要做轻量自维护。

默认巡检时机：

1. 每次高频问答结束后
2. 每次执行窗口结果回收后
3. 每次任务状态变化后
4. 每次准备继续派工前

默认巡检动作：

1. 先看 `指挥官当前任务卡.md` 是否需要改状态、改下一步最小动作或触发归档
2. 再看这轮结果是否改变了稳定事实，需要同步到本文档
3. 再看是否形成了新的高频解释，需要同步到 `指挥官问题索引.md`
4. 最后看是否形成了阶段性完成节点，需要同步到 `指挥官时间线.md`

上下文压缩前的最低写回要求：

1. 当前活跃主线处于哪一阶段
2. 最近一次可信完成项是什么
3. 下一步最小动作是什么
4. 是否存在阻塞或待用户决策

如果这 4 项没有写回外部真相源，视为“状态尚未真正交接完成”。

默认预警阈值：

1. 任命文档一旦开始影响“单文件快速启动”，就优先压缩而不是追加
2. 当前任务卡活跃任务超过 3 条时，先收缩优先级，不继续扩张
3. 同一活跃任务连续 3 轮无状态变化时，必须重写最小动作或改状态
4. 问题索引同主题重复堆积时，合并成标准解释

一句话说：

**指挥官系统要靠“固定时机巡检 + 触发式压缩”保持可用，而不是等文档失控后再补救。**
