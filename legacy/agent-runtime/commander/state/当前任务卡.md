# 指挥官当前任务卡

更新时间：2026-04-14

## 1. 文档定位

这份文档只记录“当前仍然活跃、还需要继续推进”的任务。

它不是：
- 时间线
- 问题索引
- 完整历史档案

如果只是想快速知道“现在还有什么没收口”，先读这份文档。
如果要看已经完成的阶段事实，读 `commander/state/时间线.md`。

---

## 2. 使用规则

1. 这里只保留当前仍然活跃的任务，尽量控制在 1 到 3 条。
2. 每条任务必须写清：
   - 目标
   - 当前状态
   - 已完成到哪里
   - 当前里程碑 / phase
   - 当前推进段
   - 恢复锚点
   - 阻塞点
3. 对于长任务，`恢复锚点` 只负责上下文压缩后的续跑定位，不等于完整计划，也不构成默认停机点。
4. 任务一旦真正收口，就从本文档移出，并把结果沉淀到：
   - `commander/state/时间线.md`
   - 必要时再同步到 `commander/core/主文档.md` 或 `commander/state/问题索引.md`
5. 不允许把已经进入 `done_waiting_archive` 的任务长期留在这里。
6. 如果当前没有真实活跃任务，宁可明确写“当前无活跃任务”，也不要保留过期条目。

固定状态值：
- `intake`
- `active`
- `blocked`
- `pending_user`
- `done_waiting_archive`
- `archived`

## 3. 生命周期状态与默认动作

### 3.1 `intake`

含义：
- 任务刚被识别出来
- 还没压成可执行的最小切片

默认动作：
1. 明确目标
2. 明确边界
3. 判断是否需要开执行窗口
4. 判断是否需要写入本文档

### 3.2 `active`

含义：
- 任务正在推进

默认动作：
1. 持续更新已完成部分
2. 保持“当前推进段 + 恢复锚点”始终清楚
3. 如需执行，按指挥官协议派工

### 3.3 `blocked`

含义：
- 当前任务被技术问题、验证失败或外部条件卡住

默认动作：
1. 写清阻塞点
2. 不继续机械派工
3. 先回到指挥官窗口解释为什么被卡住

### 3.4 `pending_user`

含义：
- 任务需要用户确认方向、取舍或额外输入

默认动作：
1. 写清待用户决定的点
2. 暂停继续拆下一步
3. 等用户决定后恢复到 `active`

### 3.5 `done_waiting_archive`

含义：
- 任务本身已经收口
- 但还没完成归档和沉淀动作

默认动作：
1. 把完成事实写入 `commander/state/时间线.md`
2. 必要时同步到主文档或问题索引
3. 完成后把任务从本文档移出

### 3.6 `archived`

含义：
- 任务已彻底结束
- 不再作为当前活跃任务维护

---

## 4. 状态切换规则

允许的常见切换：
- `intake -> active`
- `active -> blocked`
- `active -> pending_user`
- `active -> done_waiting_archive`
- `done_waiting_archive -> archived`

补充规则：
1. `blocked` 或 `pending_user` 恢复后，回到 `active`。
2. 不允许跳过 `done_waiting_archive` 直接把重要任务当作“自动结束”。
3. 对用户汇报“已完成 / 已推进 / 已闭环”的前提，是已经完成 `worker_report.json -> ingest -> report/status 收敛`。

## 6. 收口与归档模板

当任务进入 `done_waiting_archive` 时，指挥官默认按这个模板做最后收口：

1. 完成事实：
   - 这次真正做成了什么
2. 为什么重要：
   - 它改变了哪条规则、哪份稳定事实、哪类接班判断
3. 写入哪里：
   - `commander/state/时间线.md`
   - 如有必要，再同步到：
     - `commander/core/主文档.md`
     - `commander/state/问题索引.md`
4. 从当前任务卡移除：
   - 删除活跃条目
   - 不保留过期状态

## 7. 清理规则

如果当前活跃任务满足下面全部条件，就可以关闭并移出本文档：

1. 目标已经达成
2. 关键规则已经写入稳定文档
3. 新窗口接班不再依赖当前任务的临时说明
4. 后续只需要按维护规则增量更新，而不是继续集中治理

关闭后动作：
1. 在 `commander/state/时间线.md` 里保留完成节点
2. 删除本文档里的活跃条目
3. 如果关闭后又出现新问题，再开新的活跃任务，不复用过期状态

## 8. 例行巡检清单

默认在下面 4 个时机巡检当前任务卡：

1. 每次执行窗口结果回收后
2. 每次高频问答结束后
3. 每次任务状态切换后
4. 每次准备继续开新窗口前

巡检时按这个顺序看：

1. 当前活跃任务是不是真的仍然活跃
2. 下一步最小动作是不是足够小、足够清楚
3. 有没有任务已经该转成 `blocked` / `pending_user` / `done_waiting_archive`
4. 有没有任务已经完成却还挂在这里

默认预警阈值：

1. 活跃任务超过 3 条时，先收缩，不继续加新条目
2. 同一任务连续 3 轮没有新增完成项或状态变化时，必须改写最小动作或切状态
3. 进入 `done_waiting_archive` 的任务，不应长期停留到下一轮之后还不归档
