### 2.12 短确认词为什么会接错上一句

最短结论：

- 这通常不是事实幻觉，而是短确认词绑定错误

- 目标是让短确认词优先绑定最近一次 assistant 提议，而不是回跳到更早主题

### 2.14 为什么 RAGAS 会波动，但还要继续用，还要把样本补到 100 条

最短结论：

- retrieval eval 稳，是因为它评的是检索命中和排序这类更确定的指标
- RAGAS 会波动，是因为它让 LLM 裁判去评“答案是否忠实、是否真正回答问题”
- 但它仍然值得保留，因为它在评 retrieval eval 根本评不到的那一层

补充：

- 当前 answer-level eval 的波动，主要集中在 `faithfulness` 和 `answer_relevancy`
- 13 条样本时，单条 case 的波动会被整体均值严重放大，所以很难拿总分下强结论
- 把 paired 样本从 13 条扩到 100 条后，RAGAS 仍不会变成绝对稳定，但会更适合看趋势、分类均值和版本前后变化
- 当前主评估集已经扩成 100 条 paired case，`rag_golden_test_set.json` 和 `rag_answer_eval_test_set.json` 共享同一批 `case_id`
- 这 100 条主集优先覆盖 `faq / maintenance / troubleshooting / hard`，而不是把大量非 RAG service 问题混进 paired 主集

### 2.17 为什么不能只靠高单步成功率去堆长链自动化

最短结论：

- 因为长链路会累计误差
- 单点准确率高，不等于端到端可靠

补充：

- 如果单步成功率是 95%，连续 20 步的理想串联成功率大约只有 `0.95^20 ≈ 35.8%`
- 所以问题不只是“把每一步 prompt 再写好一点”，而是要减少纯文本链路长度，并把关键节点变成：
  - 结构化任务包
  - 结构化结果回收
  - 中途阻塞回指挥官
  - 可验证检查点
- 这也是为什么当前系统往前推进时，真正缺的是 dispatch / ingest transport 层，而不是继续叠更多长提示词

### 2.21 为什么 PowerShell / 中文路径 / 终端乱码问题总在反复出现，标准处理是什么

最短结论：

- 因为之前只沉淀了“UTF-8 重读”和“中文写入安全”，还没有把 Windows 终端侧的稳定工作规律升级成明确硬规则

补充：

- 在当前 PowerShell 工作流里，不默认依赖 `&&` 串联多个步骤；优先一工具一命令，或改用 PowerShell 原生顺序执行
- 读取中文路径文件时，不要在内联 Python 里直接手写中文文件名；先用 `Get-ChildItem`、glob 或其他目录枚举方式拿到精确路径，再读取
- 终端输出乱码时，不要直接把它等同于文件损坏；先用 `Get-Content -Encoding UTF8`、`rg`，或 `py -X utf8` 重读做文件内容级确认
- 如果只是验证“某段文字是否真的写进文件”，优先用 `rg -n` 或 `Get-Content -Encoding UTF8` 做二次确认，不要依赖 Python 在终端里直接打印中文
- 以后这类问题不再视为临场经验，而是视为指挥官工作记忆的一部分；出现相同症状时，默认直接走这套 fallback，而不是重新试错

### 2.22 为什么执行窗口说“已完成”，`commander_ingest.py` 还是可能拒绝回收

- 因为人读汇报和 machine-readable report 不是一回事；只要 `worker_report.json` 还没满足 schema 合同，就不能把它当成真正可回收的完成结果

补充：

- 当前真正的收口依据始终是：`worker_report.json -> commander_ingest.py -> report.json / status.json 收敛`
- 执行窗口即使已经跑完测试、写了总结，如果 `worker_report.json` 里仍有 schema 不接受的字段值、漏填必填字段，或残留 dispatch draft 标记，ingest 都应该拒绝继续 close/archive
- 已出现过的真实案例是：`verification[].result` 写成了自定义值 `passed_with_existing_line_ending_warnings`；虽然人能看懂，但不在 schema 允许的 `passed / failed / skipped` 枚举里，所以必须先修正成合同内值，再做 ingest
- 默认处理顺序应该是：
  1. 先看 `worker_report.json` 是否 schema-valid
  2. 再看 draft metadata 是否已清理
  3. 最后才允许把“验证通过”写入 runtime 闭环
- 以后遇到“结果看起来对，但 ingest 过不去”的情况，优先排查 report 合同，而不是先怀疑 runtime 生命周期本身坏了
