# 当前任务同步器设计

更新时间：2026-04-17

## 1. 背景

当前 `ai-coding-commander` 已经具备：

- 项目内 `.codex` 骨架 bootstrap
- `当前任务.md` 作为主任务真相源
- 写回纪律
- 验证 gate
- 恢复入口

但当前体系仍然存在一个明显缺口：

**长任务执行过程中，任务状态不会自动同步回 `当前任务.md`。**

现在的真实状态是：

1. 协议和模板已经存在；
2. 写回纪律已经明确；
3. 但执行链中还没有一个轻量机制负责在关键节点更新任务状态。

结果是：

- 长任务容易状态漂移；
- `当前任务.md` 可能落后于实际执行；
- 恢复时容易读到过时信息；
- “当前任务是真相源”在长任务中还没有被真正执行到位。

本设计的目标，就是补上这一层：

**为 commander 增加一个轻量的当前任务同步器，在长任务执行的关键节点按事件更新 `当前任务.md`。**

## 2. 目标

### 2.1 本轮目标

1. 提供一个可直接调用的任务同步脚本。
2. 让 `当前任务.md` 可以在长任务执行过程中被事件驱动地更新。
3. 只更新少量稳定字段，不引入第二套任务协议。
4. 保持 `.codex/docs/当前任务.md` 继续作为唯一主任务真相源。
5. 为后续更强的任务编排留出接口，但本轮不引入重型 runtime。

### 2.2 非目标

本轮明确不做：

1. 不做后台 daemon。
2. 不做自动监控文件变化的常驻进程。
3. 不做复杂任务图编排。
4. 不做自动推断所有任务字段。
5. 不引入 `.codex-tasks/`、`TODO.csv` 或新的主任务协议。
6. 不替代 `当前任务.md`，只增强它的可维护性。

## 3. 设计原则

### 3.1 事件驱动，不做常驻进程

同步器只在明确的执行节点被调用：

- 任务开始
- 阶段切换
- 验证完成
- 收口前

而不是后台持续运行。

### 3.2 当前任务卡仍然是唯一主真相源

同步器的职责不是创建新的任务状态系统，而是：

**把执行过程中的关键信息稳定写回 `当前任务.md`。**

### 3.3 只更新稳定字段

同步器第一版只允许更新那些：

- 结构稳定
- 容易验证
- 不容易引发歧义

的字段。

### 3.4 允许人工判断，避免伪智能

第一版不强求自动推理复杂任务语义。

同步器接收明确输入，再写回文档；
不伪装成“自己已经懂了一切”。

## 4. 同步对象

同步目标固定为：

- `.codex/docs/当前任务.md`

第一版不修改：

- `当前状态.md`
- `验收记录.md`
- `归档索引.md`

这些仍然按现有 write-back 纪律由 commander 或项目自身在合适时机更新。

## 5. 要更新的字段

第一版支持以下字段：

1. `当前进度`
2. `当前卡点`
3. `验证状态`
4. `验证证据`
5. `下一步`
6. `最近验证`

说明：

- 这些字段都已经存在于标准模板中；
- 不要求第一版修改任务名称、任务目标、任务模式、任务形状、执行强度；
- 这些更偏任务定义层，仍然以人工确认或上层流程控制为主。

## 6. 事件模型

第一版定义 4 类标准事件：

### 6.1 `start`

表示任务正式进入执行阶段。

典型效果：

- `当前进度` 从“未开始”进入“进行中”
- `下一步` 被刷新为当前最小执行动作

### 6.2 `phase`

表示长任务内部阶段切换。

典型效果：

- 更新 `当前进度`
- 更新 `当前卡点`
- 更新 `下一步`

### 6.3 `validate`

表示一次验证动作已经完成。

典型效果：

- 更新 `验证状态`
- 更新 `验证证据`
- 更新 `最近验证`

### 6.4 `preclose`

表示准备进入收口或 stop-gate 前的最后同步。

典型效果：

- 刷新 `当前进度`
- 刷新 `验证状态`
- 刷新 `验证证据`
- 刷新 `下一步`

## 7. 脚本形态

建议新增脚本：

- `skills/commander-mode/scripts/sync_current_task.py`

### 7.1 输入

脚本至少接收：

1. `--repo`
   - 目标项目根目录
2. `--event`
   - `start / phase / validate / preclose`
3. 可选字段输入：
   - `--progress`
   - `--blocker`
   - `--validation-status`
   - `--validation-evidence`
   - `--next-step`
   - `--last-validation`

### 7.2 输出

脚本输出 JSON，至少包括：

1. `updated`
2. `target`
3. `changed_fields`

如果目标文件不存在，输出明确错误，不静默失败。

## 8. 更新策略

第一版采用“按字段替换”的保守策略：

1. 读取 `当前任务.md`
2. 基于固定字段前缀定位目标行
3. 只替换本次明确指定的字段
4. 未传入的字段保持原值

这样做的原因：

- 模板稳定；
- 风险低；
- 便于测试；
- 不需要引入 Markdown AST 或复杂解析。

## 9. 与现有 commander 的关系

### 9.1 不替代 write-back 纪律

同步器不是 write-back 纪律的替代品，而是：

**把其中最容易漏掉的“当前任务状态更新”做成可执行工具。**

### 9.2 不替代 stop-gate

`portable_harness.py --cwd . stop-gate` 仍然负责轻量 stop-gate；
同步器只负责在 stop-gate 前把当前任务状态写回更真实的值。

### 9.3 不替代任务定义

任务名称、任务目标、任务类型、任务形状、执行强度仍然由任务定义层决定；
同步器只维护执行过程中的动态状态字段。

## 10. 推荐使用时机

后续 commander 工作流里，推荐在这些节点调用同步器：

1. 任务正式开始时
2. 长任务切换阶段时
3. 每次关键验证完成时
4. 收口前 / stop-gate 前

如果任务很短，也可以只在：

- 开始
- 验证后

调用两次。

## 11. 验收标准

实现完成后，至少应满足：

1. 能在一个已初始化项目中定位 `.codex/docs/当前任务.md`
2. 能只更新指定字段，不误改其他字段
3. `start / phase / validate / preclose` 四类事件都能正常工作
4. 目标文件不存在时有明确失败输出
5. 输出 JSON 可读且 UTF-8 正常
6. 不引入新的主任务协议
7. 不破坏现有 bootstrap / portable harness 测试

## 12. 建议实施顺序

建议按以下顺序实现：

1. 新增 `sync_current_task.py`
2. 为标准模板和字段格式写测试
3. 为 4 类事件写最小回归测试
4. 在 `commander-mode` 文档里补充调用建议
5. 视需要再决定是否把同步器接入更高层工作流

## 13. 一句话总结

本轮不是做一个新的任务系统，而是给当前 commander 补一个：

**“把长任务关键状态按事件同步回 `当前任务.md` 的轻量同步器”。**
