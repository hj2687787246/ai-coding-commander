# High-Signal Commander v2 设计

## 1. 背景

当前 `commander-mode` 已经具备项目恢复、`.codex` 初始化、当前任务卡、验证 gate、portable harness 和状态同步脚本等能力。但它的默认心智仍偏向“项目协议模板”：进入未初始化仓库时容易引导用户创建 `.codex` 骨架，长任务状态也更像阶段末尾的结果写回。

新的目标不是把它做成平台，也不是追求极低 token。新的目标是让 `commander-mode` 作为一个自动运行的高信号 skill：它应主动选择有价值的上下文、自动沉淀恢复所需的稳定事实，并在长任务中用轻量检查点避免中断后回到开始前。

## 2. 目标

1. 把 `commander-mode` 的主定位改为高信号指挥 skill，而不是项目模板协议本身。
2. 让 `.codex` 成为自动记忆面，而不是用户手动要求后才使用的记录区。
3. 引入过程型检查点：长任务中只要出现会影响恢复的稳定事实，就在继续下一段工作前写回。
4. 用价值门控控制 token 和文件噪声：不追求少读少写，而是只读写能降低不确定性、减少返工、保护验证纪律的信息。
5. 保留现有 bootstrap、sync、harness 能力，但把它们定位为可选工具和记忆 kit，而不是 skill 的使用前提。

## 3. 非目标

1. 不把本仓库升级为多项目平台、全局观测系统或重型 agent runtime。
2. 不强制所有目标仓库使用完整 `.codex` 模板。
3. 不把聊天全文、长篇过程总结或模型推理写进项目记忆。
4. 不以最低 token 为唯一目标；必要的上下文读取和设计推理仍然允许充分展开。
5. 不改变 `legacy/agent-runtime/` 的归档边界。

## 4. 核心产品定义

`commander-mode` 是一个高信号 AI coding 指挥 skill。它根据用户意图和仓库现状，选择最有价值的上下文，形成下一步决策，保护验证纪律，并自动维护最小可恢复项目记忆。

一句话原则：

> 用户负责决策，skill 负责不丢上下文。

## 5. 运行模型

每次触发 `commander-mode` 时，先走一个固定但轻量的决策循环：

1. 判断用户意图：恢复、指挥、实现、审查、收口、交接、架构判断。
2. 判断上下文投资范围：这次需要读哪些真相源才能降低不确定性。
3. 读取最少必要证据：优先当前代码、命令结果、仓库规则、任务状态和验证方式。
4. 给出当前判断：当前目标、风险、下一步最小动作、需要的验证。
5. 执行用户授权范围内的动作。
6. 自动判断是否需要写回。
7. 如果需要，写入最小稳定增量。

## 6. 上下文投资原则

token 的目标是高回报，不是绝对节省。

有价值的 token：

1. 能确认当前任务和边界。
2. 能发现仓库规则、危险路径或验证方式。
3. 能改变下一步决策。
4. 能避免重复恢复、重复排查或重复解释。
5. 能保存未来恢复必须知道的稳定事实。

低价值 token：

1. 机械读取完整模板和全部历史。
2. 把状态文件原样复述给用户。
3. 写入长篇聊天总结。
4. 在没有执行价值时生成过度计划。
5. 为了满足形式而创建或扩展项目协议。

## 7. 自动写回规则

写回不再依赖用户说“沉淀一下”。`commander-mode` 每次运行都必须判断是否出现恢复价值节点。

需要写回的事件：

1. 当前任务、目标、范围或模式改变。
2. 做出会影响后续工作的关键决策。
3. 发现或解除阻塞。
4. 完成一个可恢复的子步骤。
5. 获得验证证据或验证失败证据。
6. 即将等待长命令、子任务、用户确认或可能发生中断。
7. 用户表达稳定协作偏好。

不写回的内容：

1. 聊天原文。
2. 临时猜测。
3. 长篇过程叙述。
4. 没有后续恢复价值的中间输出。
5. 模型内部推理。

## 8. 检查点模型

长任务使用覆盖式检查点，而不是追加式日志。默认检查点不超过 8 行，每行只保留一句可执行事实。

推荐字段：

```markdown
# 当前恢复锚点

- 当前目标：
- 当前阶段：
- 最近完成：
- 当前卡点：
- 正在关注的文件：
- 下一步：
- 验证状态：
- 最近验证：
```

字段以覆盖更新为主。只有需要审计历史时，才追加到单独时间线；默认不维护长时间线。

## 9. `.codex` 的新定位

`.codex` 是自动记忆面，不是平台化前提。

最小模式：

1. 已有 `.codex` 时，优先用其中的恢复锚点和当前任务信息。
2. 没有 `.codex` 时，`commander-mode` 仍然正常工作，先基于仓库现有规则、README、docs、git 和测试恢复。
3. 当出现恢复价值节点时，skill 自动选择写回面：优先写入仓库已有任务文件，其次写入已有 `.codex`，最后才创建最小单文件恢复锚点。
4. 最小单文件恢复锚点只保存当前目标、阶段、卡点、下一步和验证证据；它不是完整 `.codex` 模板。
5. 完整 `.codex` 模板只在用户明确要长期治理、批量任务或多阶段项目记忆时使用。

完整 `.codex` 模板仍然保留，用于需要长期治理的仓库。

## 10. 工具定位

`portable_harness.py`、`bootstrap_codex_workspace.py`、`sync_current_task.py` 都是辅助工具，不是产品中心。

建议后续工具方向：

1. `status --brief`：给 skill 消费的短状态摘要。
2. `status --json`：需要深查时输出完整结构化信息。
3. `checkpoint`：覆盖式写入恢复锚点。
4. `stop-gate`：收口前验证 dirty worktree 和验证证据。
5. `doctor`：检查安装、Python、pytest、模板和 harness 是否可用。

## 11. 与现有版本的关系

保留：

1. 现有 `commander-mode` 的恢复、验证、写回、边界意识。
2. 现有 `.codex` 模板和 bootstrap 脚本。
3. 现有 portable harness 和 stop gate。
4. 现有 `sync_current_task.py` 的长任务状态同步方向。

调整：

1. `SKILL.md` 从模板中心改为意图路由、上下文投资、自动检查点和验证纪律。
2. 未初始化仓库不再默认推完整 bootstrap。
3. `.codex` 从“协议前提”变为“自动记忆面和可选增强包”。
4. 状态写回从任务结束后的总结，改为长任务过程中的恢复检查点。

## 12. 验收标准

1. `SKILL.md` 明确表达 high-signal commander 定位。
2. `SKILL.md` 明确：没有 `.codex` 的仓库也能正常使用 commander。
3. `SKILL.md` 明确：自动写回由价值门控触发，不依赖用户手动要求。
4. `SKILL.md` 明确：长任务中断前应写覆盖式检查点。
5. `.codex` 相关工具仍然可用，但文档不再把完整模板作为默认前提。
6. 测试覆盖至少包括：
   - 未初始化仓库不会被强制 bootstrap。
   - 自动检查点只更新稳定字段。
   - 检查点输出保持短小。
   - stop gate 仍然要求验证证据。
7. README 能清楚表达：本仓库交付的是 skill，不是平台。

## 13. 一句话总结

High-Signal Commander v2 的目标不是少用 token，而是让 token 花在降低不确定性、保护恢复能力、推动任务和保留关键事实上。
