# Taskmaster 能力吸收设计

更新时间：2026-04-17

## 1. 背景

当前 `commander-mode` 已经具备：

- 已初始化 / 未初始化项目分流
- 项目内 `.codex` 骨架 bootstrap
- 基于当前项目自身 `.codex` 协议恢复状态
- 任务导向而非学习导向的默认骨架

外部 `taskmaster` skill 提供了一套更重的任务执行协议，核心价值在于：

- 任务分型（Single / Epic / Batch）
- 长任务恢复依赖磁盘真相源
- 未验证不得标记完成
- 批量任务的结构化输入 / 输出协议

但 `taskmaster` 同时也自带一整套新的主协议：

- `.codex-tasks/`
- `TODO.csv`
- `PROGRESS.md`
- `EPIC.md`
- `SUBTASKS.csv`

这套主协议如果直接并入当前 commander 体系，会和现有 `.codex/docs/*` 真相源发生冲突。

因此，本设计的目标不是“安装 taskmaster”，而是：

**吸收 taskmaster 里最有价值的任务治理能力，增强现有 commander 体系，而不替换当前 `.codex` 协议。**

## 2. 目标

本轮设计目标：

1. 为当前 commander 体系增加明确的任务分型能力。
2. 为当前 commander 体系增加执行强度表达。
3. 将“未验证不得标记完成”升级为显式规则。
4. 将“磁盘真相源优先”升级为任务恢复层的标准规则。
5. 为未来批量任务保留轻量协议入口，但不将其变成默认骨架。

## 3. 非目标

本轮明确不做：

1. 不引入 `.codex-tasks/` 目录作为新的默认任务根。
2. 不引入 `TODO.csv / PROGRESS.md / EPIC.md / SUBTASKS.csv` 作为新的主真相源。
3. 不把 `taskmaster` 作为新的默认执行协议替换 `commander-mode`。
4. 不强制所有项目都启用批量任务协议。
5. 不在本轮引入新的自动任务编排 runtime。

## 4. 设计原则

### 4.1 吸收思想，不复制协议

只吸收 taskmaster 中可增强当前体系的治理能力，不复制它的完整目录结构和任务协议。

### 4.2 当前项目真相源优先

任务真相源仍以当前项目自身 `.codex` 文档为准，尤其是：

- `.codex/docs/当前状态.md`
- `.codex/docs/当前任务.md`
- `.codex/docs/验收记录.md`
- `.codex/docs/恢复入口.md`

### 4.3 任务治理增强不应打断现有恢复链

恢复流程仍由 `commander-mode` 和项目 `.codex` 协议主导。新增字段和规则只做增强，不另起一套入口。

## 5. 要吸收的能力

### 5.1 任务分型

在当前 commander 体系中引入统一的任务形状概念：

- `single`
- `epic`
- `batch`

用途：

- `single`：单一交付物、单共享上下文任务
- `epic`：多子任务、存在依赖关系的长任务
- `batch`：同模板、大量独立行任务

#### 落点

在 `.codex/docs/当前任务.md` 中增加字段：

- `当前任务形状`

默认值：

- `single`

### 5.2 执行强度

引入执行强度概念：

- `compact`
- `full`

用途：

- `compact`：短小、线性、低恢复成本任务
- `full`：需要更强恢复、验证、阶段记录的任务

#### 落点

在 `.codex/docs/当前任务.md` 中增加字段：

- `执行强度`

默认值：

- `full`

说明：

当前 commander 的默认风格应偏向 `full`，只有任务足够短小且恢复成本极低时，才降为 `compact`。

### 5.3 验证 gate

明确吸收 taskmaster 的一条核心治理能力：

**没有验证证据，不得标记任务完成。**

需要防止以下误判：

- 代码已写完 ≠ 完成
- 已分发给别的窗口 ≠ 完成
- 已执行脚本 ≠ 完成
- 用户口头认可 ≠ 技术完成

#### 落点

1. `commander-mode` 的完成判定规则
2. 项目 `.codex/AGENT.md` 的任务推进规则
3. `.codex/docs/当前任务.md` 中的验证状态字段

建议新增字段：

- `验证状态`
- `验证证据`

### 5.4 磁盘真相源优先

将 taskmaster 的恢复原则吸收到 commander：

**长任务恢复时，磁盘上的当前任务真相源优先于聊天记忆。**

这和现有体系一致，但需要上升为更明确的任务治理表达。

#### 落点

1. `commander-mode` 的恢复规则
2. `.codex/AGENT.md` 的恢复协议
3. `.codex/docs/恢复入口.md` 的长任务恢复说明

### 5.5 轻量 Batch 协议

为未来批量任务引入可选的轻量协议，但不进入默认骨架。

建议的可选目录：

```text
.codex/
  batch/
    <task-name>/
      BATCH.md
      workers-input.csv
      workers-output.csv
```

#### 设计边界

1. 只在任务形状为 `batch` 时启用。
2. 不作为所有项目都默认生成的目录。
3. 只服务于结构化、行级独立的批量任务。

## 6. 明确不吸收的内容

以下内容明确不进入当前 commander 主协议：

### 6.1 `.codex-tasks/`

不新增第二套任务根目录，避免和现有 `.codex/docs/*` 发生双协议冲突。

### 6.2 `TODO.csv / PROGRESS.md / EPIC.md / SUBTASKS.csv`

不把这些文件作为 commander 体系里的新主真相源。

如后续需要结构化任务追踪，应优先扩展现有 `.codex/docs/当前任务.md` 或其配套轻量文件，而不是整套照搬。

### 6.3 taskmaster 作为默认执行协议

`commander-mode` 继续作为当前主执行 / 恢复协议。

taskmaster 只作为思想来源，不作为默认外部依赖。

## 7. 与现有 commander 的关系

本设计与当前 commander 体系的关系是：

- `commander-mode`：继续负责项目识别、恢复、推进、验收、收口
- 项目 `.codex` 协议：继续负责当前项目自身的长期状态
- 新吸收能力：只增强任务治理表达，不替代当前协议

简化后的职责变化：

### 之前

- 有当前任务，但任务形状与强度不够清晰
- 有验证要求，但未形成统一任务字段
- 有恢复规则，但任务恢复表达不够显式

### 之后

- 当前任务具备“形状 + 强度 + 验证状态”
- 长任务恢复规则更明确
- 批量任务有轻量扩展入口
- 仍保持 `.codex` 单主协议

## 8. 建议实施顺序

建议后续实现按以下顺序推进：

1. 扩展 `.codex/docs/当前任务.md` 模板字段
2. 调整 `commander-mode`，吸收任务形状 / 执行强度 / 验证 gate 表达
3. 调整项目 `.codex/AGENT.md` 与 `恢复入口.md`
4. 仅在需要时再补轻量 batch 模板

## 9. 验收标准

后续实现完成后，应满足：

1. `commander-mode` 不需要引入 `.codex-tasks/` 也能表达任务分型。
2. 当前任务卡中可明确区分：
   - 任务形状
   - 执行强度
   - 验证状态
3. 完成判定不再依赖口头描述，必须有验证证据字段。
4. 长任务恢复时，优先读取项目内任务真相源，而不是回忆聊天。
5. 在没有批量任务时，不会平白出现新的批量目录噪音。

## 10. 一句话结论

**taskmaster 最值得吸收的是任务治理能力，而不是它整套新的任务协议；当前 commander 应该增强，而不是替换。**
