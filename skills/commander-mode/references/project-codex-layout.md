# Project `.codex` Layout

这份参考文档只解决一个问题：

**一个项目里，哪些规则应该放全局 skill，哪些应该放项目内 `.codex`，哪些应该留在项目正文里。**

## 1. 四层结构

推荐把 AI coding 相关内容分成四层：

1. 全局 skills
2. 项目 `.codex` 工作区
3. 项目正文文档
4. MCP 资源层

它们不是同一个东西。

## 2. 全局 skills

位置：

`C:\Users\<user>\.codex\skills\`

放什么：

- 跨仓库可复用的能力
- 调试、排障、编码规范、commander、review、UTF-8 IO 之类的通用 skill

不要放什么：

- 某个具体项目的学习计划
- 某个仓库的恢复状态
- 某个项目独有的业务规则

判断标准：

如果一个能力在多个仓库都应该复用，就放全局 skills。

## 3. 项目 `.codex` 工作区

位置：

`<repo>/.codex/`

放什么：

- 当前项目的 Codex 工作说明
- 当前项目的当前状态与当前任务
- 当前项目的验收记录
- 当前项目的恢复入口
- 当前项目的归档与偏好索引
- 当前项目专属编码规范、注释规范、日志规范
- 当前项目内 harness 的边界和最小用法

推荐结构：

```text
<repo>/
  AGENTS.md
  README.md
  .codex/
    AGENT.md
    docs/
      当前状态.md
      当前任务.md
      恢复入口.md
      验收记录.md
      归档索引.md
      协作偏好.md
      周总结.md
```

说明：

- 默认骨架使用 `当前任务.md`，不是学习进度卡。
- 学习模式只是任务模式之一，不是通用 commander 协议的默认模式。
- 只有项目自己进入长期学习模式时，才需要额外长出学习进度卡、学习时间线、学习主线等文档。

判断标准：

如果内容只对当前项目有意义，就放 `.codex/`，不要升级成全局 skill。

## 4. 项目正文文档

位置：

`<repo>/README.md` 和 `<repo>/docs/`

放什么：

- 项目介绍
- 架构说明
- 运行方式
- 接口说明
- 部署说明
- 排障文档
- 面向开发者或面试表达的项目资料

不要放什么：

- 长篇 AI 协作规则
- 通用 commander 方法论
- 项目学习过程管理

判断标准：

如果这是“项目本身是什么、怎么运行、怎么理解”，就放项目正文。

## 5. MCP 资源层

MCP 不是目录结构，而是 Codex 访问外部资源的通道。

可以这样理解：

- skills：教 Codex 怎么做
- `.codex`：告诉 Codex 在这个项目里该怎么工作
- MCP：给 Codex 提供额外资源或上下文来源

MCP 是否可用，取决于 Codex 客户端和当前环境的 server 配置，不等于每个仓库默认自带自己的 MCP。

## 6. 根目录 `AGENTS.md` 应该多薄

推荐把根目录 `AGENTS.md` 收成门牌文件，只保留：

1. 语言和编码硬约束
2. 仓库主身份
3. 第一跳入口
4. 不要乱动的高风险路径

详细项目规则、学习主线、注释规范、日志规范，统一放到 `.codex/AGENT.md`。

## 7. 默认落地规则

以后新项目可以按这套判断：

- 所有项目都能复用的能力：放全局 skill
- 只对当前项目有意义的协作规则：放 `.codex/`
- 项目本身的功能和文档：放 `README.md` / `docs/`

先做项目内 `.codex`，只有当某套规则在多个项目里反复出现时，再把它升级成全局 skill。
